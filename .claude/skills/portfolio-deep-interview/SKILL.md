# 포트폴리오 심층 인터뷰

## 개요

심층 인터뷰는 **코드가 말해주지 못하는 것**을 캐낸다: 왜 그 대안을 버렸는지, 어떻게 검증했는지, 어떻게 배포했는지, 어디까지가 본인 몫이었는지. 코드 정찰(2단계)이 이미 코드로 아는 칸을 채웠으니, 네 일은 나머지를 — *방어 가능하게* — 채워 케이스 스터디가 주장이 아니라 증거로 읽히게 하는 것이다.

**핵심 원칙: 너는 폼 감사자가 아니라 면접관이다. 인터뷰 한 턴은 한 주제를, 깊게, 프라이버시 계약을 한 번만 선언하고, 얕은 답은 어필급 신호가 나올 때까지 되판다.** 질문을 한꺼번에 쏟으면 얕은 답이 나오고, 얕은 답은 "주장처럼" 읽히는 케이스 스터디를 만든다.

**시작 전 읽기:** case-study-structure.md §9.1(원자료 폼)·§9.2(루브릭)를 먼저 읽는다. 읽을 수 없으면 그 섹션을 안다고 가정하지 말고, 아래 **주제 마감 체크리스트**만 근거로 쓴다.

## 자세 — 폼 감사자가 아니라 면접관

칸을 다 채우는 게 목표가 아니다. **면접관이 보고 꽂힐 신호를 끌어내는 것**이 목표다. case-study-structure.md가 정의한 어필 = 시니어 시그널(근본원인 추적·트레이드오프 인식·버린 선택지[P4]) · 반증 가능한 디테일(해본 사람만 아는 구체성[P5]) · 명명 가능한 기술 메커니즘 · 방어 가능한 결과. 질문은 이걸 *끄집어내도록* 설계한다.

**금덩이를 놓치지 마라 (nugget pounce).** 답에 강한 신호가 *평평하게* 박혀 있으면(예: "공식 버그를 인지하고 패치 아닌 교체를 판단", "14,000줄을 안전하게 걷어냈다", 트레이드오프를 입에 올림) — 그게 *가장 강한 어필 후크*임을 알아채고 **거기를 판다**. 칸 체크하고 넘어가는 게 가장 큰 실패다. 평평한 한 줄을 헤드라인·콜아웃급 디테일로 끌어올리는 게 면접관의 일이다.

**끌어낸 신호를 케이스 층에 매핑하며 묻는다** (단, 아래 예시 질문은 *사용자가 그 사실을 이미 말했을 때만* 쓴다 — "유도 금지" 참조):

| 어필 신호 | 케이스 층 (case-study-structure.md) | 끌어내는 질문 |
|---|---|---|
| 명명 가능한 기술 메커니즘 | ① 헤드라인 | "이걸 한 줄 메커니즘 이름으로 부른다면?" |
| 해본 사람만 아는 구체성 (재현조건·충돌지점) | ③ 핵심 콜아웃 | "그 충돌 지점을 단계로 — 해본 사람만 알 수준으로" |
| 근본원인·트레이드오프·버린 선택지 | ④ 본문 지배질문 (시니어 시그널) | "패치 두고 교체한 판단 근거 + 그 비용은?" |
| 방어 가능한 결과·제거한 위험 | ② 요약표 R행 | "14k줄 안전 청산을 어떻게 보장했나(grep/테스트)?" |

## 인터뷰 한 턴 — 레시피 (순서대로)

좋은 인터뷰는 이 턴의 연속이다. 각 턴은:

1. **한 턴에 한 주제.** 다주제 케이스(예: 딥다이브 5개)? 주제별로 하나씩, 모든 주제 × 모든 칸을 한 메시지에 펼치지 마라(베이스라인 1순위 실패). **시작 순서**: ① 사용자/운영 피해가 가장 컸던 문제 → ② 기술 판단이 가장 갈렸던 결정 → ③ 검증/마이그 리스크가 가장 컸던 작업 → ④ 본인 기여가 가장 선명한 작업. 동률이면 *버린 대안이 있는* 주제부터.
2. **user-only 칸만 묻는다.** 코드 정찰이 이미 코드로 아는 칸을 채웠다 — 재질문하지 마라. 정찰 사실을 먼저 보여주고, 그 위에 why/how를 얹어달라고 한다.
3. **구체성을 강제한다.** 실패 체인을 *단계 시퀀스*로 요구한다 — "X를 하면 → Y가 깨지고 → 사용자에게 Z가 보인다" — 추상("RHF가 한계였다", "복잡도를 개선했다") 금지. 추상답은 거부하고 단계를 다시 묻는다.
4. **probe 축을 바꿔가며 마감 체크리스트까지 되판다.** 답이 얕으면 *같은 각도로 반복하지 마라* — 축을 바꾼다: 1차 실패 체인 → 2차 버린 대안/비용 → 3차 검증/증거. 2번째 probe부터 이유를 한 줄 붙인다("이 부분이 케이스 신뢰도를 좌우해서 한 번만 더 좁힐게요"). **정지 조건**: 마감 체크리스트 통과 / "모름" / 3축 다 돌림 — 그 다음 칸을 닫는다. 3축 다 돌려도 안 나오면 **"근거 약함"으로 표시하고** 넘어간다(무한 추궁 금지). (단발로 끝내면 평평한 답에서 멈추고, 같은 축만 반복하면 헛돈다.)
5. **"모름 / 기억 안 남"은 유효한 답이다.** 추측으로 빈칸을 메우지 마라 — 포트폴리오를 거짓으로 만든다. 원자료엔 확실성 태그를 단다: **[확정]**(사용자가 말함·방어가능) / **[일반화]**(기밀이라 표현만 바꿈) / **[미확인]**(정찰·추정, 사용자 확인 전) / **[빈칸]**(모름). **[미확인]은 케이스 본문 근거로 쓰지 않는다.**
6. **주제를 닫고 전진한다.** 결과 / 남은 비용은 그 주제의 사실이 다 모인 뒤 마지막에 합성한다.
7. **매 턴 끝에 진행상황을 보고한다.** 응답 마지막에 진행 상태 블록을 붙인다(→ §진행상황 보고). 사용자가 "어디까지 왔나"를 묻기 전에 늘 보이게 한다.

## 진행상황 보고 — 매 턴 끝에 (필수)

매 응답 **마지막**에 아래 **표**를 붙인다(한 줄 압축 ❌ — 한눈에 스캔되는 표 ⭕). 사용자가 묻기 전에 항상 보인다 — 인터뷰는 여러 턴에 걸쳐 길어지므로, 현재 위치와 *끝 조건*이 매 턴 보여야 사용자가 범위를 통제할 수 있다.

| 토픽 | 상태 |
|---|---|
| ① <토픽명> | ✅ 닫음(원자료 기록 완료) |
| ② <토픽명> | ✅ 닫음 |
| ③ <토픽명> | ⏳ 진행 중 — <한 줄 메모> |
| ④ <토픽명> | ☐ 다음 |
| ⑤ <토픽명> | ☐ 대기 |
| 관통 테마 합성(AI네이티브·§0 정정 등) | ☐ 끝에 합성 |

표 바로 아래 한 줄: **지금** \<현재 토픽\> · **다음** \<다음\> · **끝 조건** \<남은 것\> → §9.1 합성·기록 → 3단계 종료

- 토픽 목록은 케이스마다 다르다(딥다이브 N개에 맞춤). 기호: **✅ 닫음 / ⏳ 진행 / ☐ 대기·다음**. 상태 칸엔 짧은 메모를 괄호로(예: "닫음(파일 기록 완료)", "닫는 중 — 통합 디테일만").
- 토픽을 열거나 닫을 때 상태를 갱신한다. 끝 조건(무엇이 남았고 어디서 끝나는지)은 표 아래 한 줄로 매번 명시 — "어디까지 질문하나"가 늘 보이게.
- 사용자가 더 빨리 끝내고 싶을 수 있으니, 남은 분량이 많으면 "몰아서 받기" 옵션도 제시한다.

## 유도 금지 — 안 한 말을 넣지 마라

사용자가 말하지 않은 메커니즘·숫자·대안을 질문에 *심지 마라.* 면접관이 스토리를 먹여주면 어필이 아니라 *조작*이 된다 — 비공개 작업물에서 가장 치명적인 실패다.

- 나쁨(유도): "패치 대신 교체한 이유는?" ← 사용자가 '교체'를 말 안 했으면 답을 심는 것
- 좋음(중립): "당시 고려한 선택지는 뭐였고, 최종안을 고른 결정적 기준은?"
- 원인·수치·의도를 제안하지 마라. "혹시 X였나요?"가 아니라 "당시 판단 기준은 무엇이었나요?"로 묻는다.
- **금덩이 포착은 사용자가 *이미 꺼낸* 신호에 한해 판다.** 위 매핑 표의 예시 질문도 그 사실을 사용자가 말했을 때만 쓴다.

## 프라이버시 계약 — 첫 주제 전에 한 번

사용자가 케이스 신뢰도에 필요한 *비식별 구조 정보*까지 "기밀이라서"로 통째로 비우지 않게, 일반화 경로를 먼저 제시한다:

- **정당한 기밀 → 일반화** (차단 X): 트래픽 수치·장애 건수·이탈률 → "대형 종합병원 환경", "피크 시간대"로.
- **케이스 신뢰도에 필요한 비식별 구조 정보**: 정확한 값이 기밀이면 **범위·비율·상대 규모**로 — 화면명 대신 "몇 종류의 플로우", 정확한 테스트 수 대신 "단위/통합/E2E 중 무엇을 몇 범주로". 일반화는 해도 *통째로 비우지는* 않는다.

## 칸 분담표 — 누가 무엇을 답하나

| 원자료 폼 칸 | 출처 |
|---|---|
| 영향 범위 정량 (화면/모듈 수) | 셀 수 있으면 코드 정찰, 아니면 **질문** |
| 문제(실패 체인) · 재현 조건 | **질문** — 코드는 증상을 보여줄 뿐, 겪은 실패는 아니다 |
| 버린 대안 + 버린 이유 | **질문** — ★최고 가치, 코드에 절대 없다 |
| 채택안 + 채택 이유 (결정적 이유) | **질문** |
| 검증 방식 (회귀: 테스트 / 호출부 점검 / QA 케이스) | **질문** |
| 배포 방식 (점진 / feature flag / QA 후) | **질문** |
| 인과 귀속 (기술 변경 덕 vs 팀 프로세스 덕) | **질문** — 과장되기 쉬움, 겸손하게 묻기 |
| 나 vs 팀 경계 | **질문** |
| 결과 (방어 가능 — Before→After 또는 상태 전환) | 사실이 모인 뒤 합성 |
| 남은 비용·후속 리스크 | **질문** — 없으면 ADR이 아니라 성공담이 된다 |
| 공개 불가 정보 (일반화할 것) | 프라이버시 계약에서 플래그됨 |

## 주제 마감 체크리스트 — 다 ✅여야 닫는다

"가장 강한 신호가 나왔나"는 vibes다. 주제를 닫기 전 아래를 전부 확인한다(못 채운 칸은 열어둔다):

- [ ] 문제/실패가 **단계 체인으로 재현** 가능한가?
- [ ] 왜 중요했나가 **사용자·운영·개발 비용** 중 하나로 연결됐나?
- [ ] **버린 대안 ≥1 + 버린 이유**가 있나?
- [ ] 채택 이유가 '좋아서'가 아니라 **제약/비용/리스크 기준**인가?
- [ ] 검증이 **테스트/QA/grep/로그/리뷰** 같은 실행 가능한 행동으로 남았나?
- [ ] 결과가 **수치·상태 전환·제거된 리스크** 중 하나로 방어 가능한가?
- [ ] **나 vs 팀** 기여가 분리됐나?

마감 직전 **회의적 면접관 3질문**을 던진다(못 버티면 칸을 열어둔다):
- "취향 리팩터가 아니라 실제 문제 해결이었다는 증거는?"
- "그 결과가 이 변경 *때문*이라고 말할 근거는?"
- "팀 성과 중 본인 몫은 어디까지인가?"

체크리스트·3질문을 통과 못 한 칸은 열린 채로 둔다 — 우회해서 쓰지 마라.

## 흔한 실수

| 실수 | 교정 |
|---|---|
| 모든 주제 × 모든 칸을 한 메시지에 쏟기 | 한 턴에 한 주제, 뼈대부터 |
| 코드로 아는 사실(LOC, 테스트 수) 재질문 | 정찰 사실을 보여주고 그 위 why/how만 묻기 |
| "RHF가 한계였다" / "복잡도 개선" 수용 | 추상 거부, 단계별 실패 체인 요구 |
| 이유 없는 결론 수용 | 넘어가기 전 "왜 그 대안은 아니었나?" 그 자리에서 되묻기 |
| "기밀이라서"가 재현/테스트/화면 수를 가리게 둠 | 프라이버시 계약 먼저: 그것들은 숨김 금지 |
| 추측으로 빈칸 메우기 | "모름"은 유효 — 빈칸 두고 넘어가기 |
| 남은 비용·리스크 건너뛰기 | 항상 묻기 — 없으면 글이 성공담이 된다 |
| 진행상황을 안 알려줘 사용자가 "어디까지 왔나" 매번 묻게 함 | 매 턴 끝에 진행상황 블록 필수(→ §진행상황 보고) |

## 쓰지 않을 때

- 2단계(코드 정찰)가 아직 안 됐을 때 — 코드로 아는 칸을 먼저 채워라. 안 그러면 코드가 답할 걸 사용자에게 물어 시간을 낭비한다.
- 인터뷰가 끝나고 MDX를 쓰는 중일 때 — 그건 case-study-structure.md의 일이지 이 스킬의 일이 아니다.

## 1. 목적·범위

- **무엇을 지배하나**: 프로젝트 상세 페이지(MDX) 한 편의 골격·작성 규칙. 홈/경력 섹션의 메타는 [`spec.md`](./spec.md) IA를 따른다.
- **왜 이게 중요한가**: 면접관은 정독하지 않는다. 구조는 **"읽힐 가능성"의 상한**이다 — 스캔에 어필 못 하면 깊은 내용도 안 읽힌다. 단 **"품질"의 상한은 증거·판단의 질**이다. 이 문서는 구조(§4)를 보장하고, 증거·판단은 **원자료 폼(§9)·루브릭(§9)**이 받친다. 구조만으론 좋은 글이 보장되지 않는다.
- **적용 단위**: 대표작(Selected Project) 1편 = 이 골격 1회. 다른 프로젝트도 **같은 골격**으로 작성한다.

## 3. 배치 모델 (어떤 작업을 상세 글로?)

- **기능 단위로 선별한다.** 제품(예: "PRO PC 앱") 단위 ❌ → 기능·문제 단위(예: "에러 자가복구") ⭕. 제품 단위는 헤드라인 훅이 죽고 lede가 묻힌다.
- **대표작은 깊은 상세 글, 나머지는 경력 불릿.** 깊이 > 개수. 강한 2~3편이 얕은 5편을 이긴다.
- **선별 기준**: ① 서로 다른 시니어 역량 차원을 덮나 ② 의사결정·오너십이 살아있나 ③ (있으면) 정량 임팩트. 숫자는 가점이지 입장권이 아니다.

## 4. 상세 글 5층 골격 (본체)

상세 글은 **위에서 아래로 점점 깊어지는 5층**이다. 스키머는 위 2~3층에서 멈춰도 핵심을 얻고, 관심 생긴 면접관만 아래로 내려간다.

```
① 헤드라인 블록   ← 5초: 이게 뭐고 왜 만만찮나
② 요약 표(PAAR)   ← 5초: 문제·판단·실행·결과 한눈에
③ 핵심 원인 콜아웃 ← 10초: "해본 사람만 아는" 반증 가능한 디테일 1조각
④ 본문            ← 30초~: 케이스별 지배 질문 1개로 판단 근거 서술
⑤ 딥다이브(항상 노출) ← 드릴다운: 증명·트러블슈팅·코드 (위치·소제목으로 자연 분리, 접지 않음)
```

각 층은 **역할이 분리돼 같은 사실을 반복하지 않는다.** 표=압축 요약 / 콜아웃=원인 1조각 / 본문=판단 근거 — 같은 문장을 세 번 재탕하면 층이 무너진다. 각 층을 분해해 정의한다.

### ① 헤드라인 블록

- **정의**: 글 진입부. 컨텍스트 칩 + 제목 + (선택)한 줄 부제.
- **기능·역할**: 도메인 무지 독자도 5초에 "무슨 기술 주제 + 만만찮은 난이도"를 잡게 한다. 후킹은 여기서 끝난다.
- **구성**
  - **컨텍스트 칩**(작게): 제품이 뭔지 1줄. 도메인 grounding. 예) `🩺 환자용 AI 문진 웹 · 사내 비공개`
  - **제목**: `[기술 토픽 키워드] + [명명된 메커니즘]`. (역피라미드 P2 + 명명된 메커니즘 P3)
  - **제목 하이라이트**(`titleHighlight` frontmatter): 제목 중 **명명된 메커니즘 구간**을 지정해 스윕 하이라이트를 입힌다. 토픽 일반어·부제(`RHF→…` 류)·결과숫자가 아니라 **메커니즘 명사구**(예: `Sibling-scoped Fractional Rank`·`불변 상태 트리`). ⚠ **`title` 안에 그대로 들어 있는 부분문자열**이라야 적용(없으면 무강조 폴백). 글을 쓸 때 제목과 함께 **반드시 정한다**(빼먹으면 후크가 못 잡는 빈칸 — 작성자가 챙길 책임).
  - **부제**(선택): 특정 기법을 익숙한 상위 개념으로 추상화해 도메인 앵커 제공(Netflix식 "QoS").
- **작성 규칙**
  - 제목에 **미스터리·스토리 후크 금지**. 명명된 기술 메커니즘 자체가 후크다.
  - **숫자**: 클릭베이트성 결과 숫자는 금지(예: '에러 37% 감소'). 단 **메커니즘과 본질적으로 결합된 숫자는 허용**(예: '렌더 40ms 줄인 가상 스크롤', '14,000줄 폼 엔진 교체'). 판단 기준 = 그 숫자가 *무엇을 했는지*를 말하면 OK, *얼마나 좋아졌는지*만 말하면 R행/부제로.
  - **메커니즘 이름은 한 일에 정직하게.** 평범한 작업에 거창한 이름(예: '조건 분기 정리'를 '상태 전이 복구 레이어'로) 붙이면 신뢰가 아니라 포장이 된다.
  - 도메인 전문용어(의료·문진 등)는 칩으로 grounding, 기술 용어(폴링·동시성)는 그대로 노출(엔지니어 in-group 언어).
- **좋은 예**: `AI 문진 에러 자가복구 — 공통 폴링을 비동기로 다시 설계`
- **나쁜 예**: `다음을 눌러도 아무 일도 일어나지 않았다`(미스터리 후크·도메인 grounding 0) / `문진 개선 작업`(작업 나열) / `에러를 50% 줄인 리팩터링`(결과 숫자를 제목에) / `상태 전이 복구 레이어`(평범한 작업 과대포장)
- **근거**: 역피라미드(NN/g) · 명명된 메커니즘 제목(Netflix "Enhancing…Prioritized Load Shedding", Stripe "Sorbet: …") · 클릭베이트 금지(HN)

### ② 요약 표 — PAAR

- **정의**: 글 전체를 4행으로 압축한 라벨형 표. **PAAR** = Problem · Alternatives(대안·근거) · Action · Result.
- **기능·역할**: 5초 스캔. 표만 봐도 "문제 → 판단 → 실행 → 결과"의 인과가 잡힌다. 줄글(설명문) ❌, 라벨형 ⭕.
- **PAAR 각 행의 정의** (라벨만 빌리지 말 것 — 의미를 지킨다)
  - **P 문제**: 무엇이 왜 문제였나. 실패 체인으로.
  - **A 대안·근거**: ★가장 중요. **버린 선택지 + 왜 버렸나.** 결론만 쓰면 PAAR이 아니라 재라벨링이다. (P4)
  - **A 실행**: 핵심 동작. 구현 수단 하나에 기대지 말 것(예: "setInterval→setTimeout"만 ❌ → 동작의 본질을 함께).
  - **R 결과**: **방어 가능한 표현으로.** 정량 있으면 `Before → After`. 없으면 **상태 전환**("먹통 → 자가복구")·**제거한 위험**("요청 중첩 가능성 제거")으로. (P5)
- **메타 줄**: 표 아래 한 줄 — `역할·스택·신뢰 고지(나 vs 팀, 비공개 일반화)`.
- **작성 규칙**
  - **굵은 글씨**로 기술명·핵심 결과 강조.
  - 결과 행에 **검증 불가능한 과장 금지**("에러 인지 0", "모든 화면 버그 제거"). 측정 정의 없으면 빼거나 방어 가능한 말로.
- **좋은 예 / 나쁜 예**

  | 행 | 나쁜 예(재라벨링·과장) | 좋은 예(PAAR 충실) |
  |---|---|---|
  | A 대안 | 공통 폴링을 비동기로 재설계 | 화면만 우회하면 **다른 화면에 동일 리스크 잔존** → 공통 폴링 재설계 |
  | R 결과 | 사용자 에러 인지 **0**, 모든 화면 버그 제거 | 이전 화면 복귀 없이 **자동 재시도** · **요청 중첩 케이스 제거** · 동일 훅 화면까지 **회귀 검증 확장** |
- **근거**: PAAR(searchright, 한국 채용 SaaS) · 스캔 가능성 +124%(Morkes & Nielsen)

### ③ 핵심 원인 콜아웃

- **정의**: 본문 앞, 시각적으로 강조된 1문단. 이 케이스의 **반증 가능한 디테일 1조각**.
- **기능·역할**: 비공개라 링크·스크린샷이 없으니, **"해본 사람만 쓸 수 있는 구체성"**이 신뢰의 첫 증거가 된다. 접지 않는다(스키머가 보는 마지막 층).
- **작성 규칙**
  - 원인 분석이면 제목을 **"핵심 원인"**, 관찰·로그·재현이면 **"핵심 증거"**로 정직하게. (둘을 혼동해 "증거"라 쓰고 원인 분석을 넣지 말 것)
  - "복잡도를 개선했다" 류 추상 ❌ → 재현 조건·충돌 지점 같은 **구체**.
- **좋은 예**: "되돌리면 방금 낸 답이 선택돼 있고 '다음'도 활성 → 다시 누름 → 답 미변경이라 더티 체킹이 서버 요청을 스킵 → 또 에러. **FE 전제와 서버 전제가 충돌한 자리.**"
- **나쁜 예**: "상태 관리를 개선해 안정성을 높였다."(반증 불가)
- **근거**: P5(반증 가능한 디테일) · F-pattern 시각 단서(NN/g)

### ④ 본문 — 케이스별 지배 질문

- **정의**: 케이스의 **가장 어려운 판단 하나**를 중심축으로 한 서술형 본문.
- **기능·역할**: 30초+ 드릴다운한 면접관에게 **판단의 질**을 보여준다. STAR처럼 모든 글을 같은 라벨로 도배하지 않는다(동질화 방지).
- **케이스별 지배 질문 변주** (제목·본문이 차별화를 담당)

  | 케이스 유형 | 지배 질문 |
  |---|---|
  | 장애·버그형 | 왜 단순 패치가 아니라 구조 문제였나 |
  | 리팩터링형 | 무엇을 보존하고 무엇을 바꿨나 |
  | 성능형 | 병목을 어떻게 특정했나 |
  | 레거시 전환형 | 한 번에 갈아엎지 않고 위험을 어떻게 쪼갰나 |
  | 협업·제품형 | 기술적으로 쉬운 선택을 왜 버렸나 |
- **본문 권장 흐름**(지배 질문 아래, 서술형 소제목으로) — **ADR 6축을 빠짐없이** 담는다(라벨만 ADR이 아니라 실제로 ADR답게):
  - **문제·제약** — 왜 이게 어려웠나 + 당시 제약 (2~3문장)
  - **핵심 결정** — 버린 선택지 + 채택안 + 왜 (PAAR의 A를 펼친 것)
  - **무엇이 좋아졌나** — 방어 가능한 결과
  - **남은 비용·후속 리스크** — 이 결정이 남긴 트레이드오프 (ADR의 Consequences — 빠지면 ADR이 아니라 성공담)
  - **어떻게 안전하게 바꿨나** — ★신뢰 장치: 영향 범위·검증 방식·배포 방식. (codex가 짚은 빈칸 — 비워두면 "빈말"이 됨)
- **작성 규칙**: 소제목은 **서술적**(단계 라벨 🚨🔎💡 금지 — 동질화·내용 0). 첫 2단어가 의미를 전달.
- **근거**: Minto(독자 질문 중심·단일 지배 논점) · ADR(제약→결정→채택→버린→결과→남은 비용 6축) · 신뢰 장치(codex)

### ⑤ 딥다이브 — 항상 노출 (접지 않음)

- **정의**: 본문 아래, **항상 보이는 최하층.** 증명·트러블슈팅·코드·버린 대안의 상세.
- **기능·역할**: 깊이를 숨기지 않는다. 스키머는 위층(①~③)에서 만족하고 **스크롤로 지나치고**, 드릴다운한 면접관만 여기까지 읽는다. 분리는 **위치(맨 아래) + 서술적 소제목**으로 한다.
- **★우리 정책: 접지 않는다 (MUST).** 접기가 *보편적* 반패턴이라서가 아니다(긴 문서에서 접기는 유효한 도구다). 다만 **이 사이트 맥락에선 접을 이득이 없다**: 상세 페이지에 길이 제약이 없고, 비공개 작업물에서 깊이가 유일한 무기인데 ① 스키머는 어차피 안 펼쳐 깊이가 죽고 ② 접힌 요약 한 줄은 안의 **서술적 소제목 스캔 면을 가린다.** 굳이 접어 얻을 게 없으니 안 접는다. (만약 접는다면 핵심 증거·payoff는 **어떤 경우에도** 접지 않는다.)
- **작성 규칙**
  - **서술적 소제목**으로 쪼갠다: "왜 setInterval을 버렸나(동시성)", "성공 후 재시도 횟수 미초기화 버그" — 스크롤 스캐너가 소제목만 보고 골라 읽을 수 있게. (layer-cake 스캔이 작동하려면 소제목이 의미를 담아야 함 — 핵심은 '접기 금지'가 아니라 '의미 있는 헤딩')
  - 길이를 줄이려 접지 말 것. 길면 소제목으로 분절할 뿐.
- **근거**: 역피라미드(만족하면 멈춤·NN/g) + layer-cake 스캔(의미 있는 서술적 소제목·NN/g) · ADR(트레이드오프).

## 5. 작성 규칙 체크리스트 (MUST)

글을 쓴 뒤 아래를 전부 통과해야 한다.

- [ ] **제목** = 기술 토픽 키워드 + 명명된 메커니즘. 숫자·미스터리·스토리 후크 없음.
- [ ] **`titleHighlight`** frontmatter가 제목의 **명명 메커니즘 구간**으로 지정됐다(토픽 일반어·부제·숫자 ❌ · `title`의 부분문자열 ⭕). 빠지면 무강조라 헤드라인 어필이 약해진다.
- [ ] **요약 표**의 A(대안) 행에 **버린 선택지 + 근거**가 있다(결론만 ❌).
- [ ] **요약 표**의 R(결과) 행이 **방어 가능한 표현**이다(측정 정의 없는 "0"·"모든" 금지).
- [ ] **핵심 콜아웃**이 반증 가능한 구체 디테일이다. 제목이 내용(원인/증거)과 일치한다.
- [ ] **본문**이 케이스별 **지배 질문 1개** 중심이다(라벨 도배 ❌).
- [ ] **신뢰 장치** 3종이 있다: ① 나 vs 팀 구분 ② 검증·배포 방식 ③ 비공개 일반화 고지. **단 메타 줄·본문에 녹인다 — 별도 "신뢰 고지" 문단으로 박지 말 것(구걸처럼 읽힌다).**
- [ ] **딥다이브가 접히지 않고 항상 보이며**, 서술적 소제목으로 분절돼 스크롤 스캔이 된다.
- [ ] **층 간 반복 없음**: 표·콜아웃·본문이 같은 사실을 재탕하지 않고 각자 역할(압축/원인/판단)을 한다.
- [ ] **숫자 위치**: 결과 숫자는 제목이 아니라 요약 표 R행·부제에 있다(메커니즘 결합 숫자는 제목 허용).
- [ ] **시각 위계**: 칩=작게 / 표=요약 / 콜아웃=강조 / 딥다이브=맨 아래(깊이). 역할이 시각적으로 구분된다.
- [ ] **포장 점검**: 메커니즘 이름이 실제 한 일에 정직한가(평범한 작업 과대명명 ❌). 반증 가능한 디테일이 그럴듯한 단어가 아니라 실제 검증 가능한가.

> ⚠️ **5층은 뼈대지 분량 균일이 아니다.** 케이스마다 긴장의 위치가 다르다 — 어떤 글은 ④본문이 길고, 어떤 글은 ⑤딥다이브가 두껍다. 모든 글을 같은 비중으로 채우면 STAR처럼 템플릿 냄새가 난다. **층은 고정, 비중은 케이스의 가장 어려운 판단을 따라간다.**

## 6. 안티패턴 — STAR가 틀린 점 (왜 이 골격이 STAR가 아닌가)

기존 STAR(🚨이슈→🔎접근→💡해결→🚀트러블슈팅→🪞회고)는 **완성도·시간순**엔 강하나 **스킴·어필**엔 역최적화다.

| STAR의 실패 | 이 골격의 교정 |
|---|---|
| 제목이 기술 묘사("상태기반→라우트기반") | 토픽 키워드 + 명명된 메커니즘 |
| outcome이 맨 뒤(역피라미드 거꾸로) | 요약 표·헤드라인으로 결론 먼저 |
| 불릿 벽·균일 가중치 → 핵심 묻힘 | 시각 위계 + 콜아웃으로 강조 |
| 단계 라벨(🚨🔎💡)이 모든 글을 동질화 | 케이스별 지배 질문으로 차별화 |
| 정량·임팩트가 본문에 묻힘 | 요약 표 R행으로 끌어올림 |
| 버린 선택지·실패담이 약함 | PAAR의 A + 딥다이브로 노출 |

## 7. 내용 확보 체크리스트 (심층 인터뷰로 채울 것)

골격은 맞아도 내용이 얇으면 "주장처럼" 보인다. 비공개 작업물은 아래를 **인터뷰로 확보**해야 한다. (회사 기밀이 아니라 문제 설명의 최소 단위 — 못 쓰면 변명)

- [ ] **영향 범위 정량** — "모든 화면"이 몇 개인가 (2개 vs 20개는 무게가 다르다)
- [ ] **검증 방식** — 회귀를 어떻게 확인했나 (테스트·호출부 점검·QA 케이스 수)
- [ ] **배포 방식** — 점진/feature flag/QA 후 배포 중 무엇이었나
- [ ] **인과 귀속** — "QA가 채택"이 이 기술 변경 덕인지, 팀 프로세스 덕인지
- [ ] **대안 근거** — 버린 선택지(화면 예외처리·무한/1회 재시도)를 왜 버렸나

> 비공개가 정당하게 가리는 것: 실제 트래픽 수·장애 건수·이탈률. **가리면 안 되는 것**: 재현 조건·테스트 케이스 수·영향 화면 수·재시도 횟수·검증 방식.

## 8. 30초 스킴 테스트 (수용 기준)

글을 **백지 상태로 30초만** 스캔했을 때 아래가 잡히면 합격.

- [ ] **무슨 기술 주제**를 다뤘나 (제목에서)
- [ ] **어떤 판단**을 했나 (요약 표 결정 행에서)
- [ ] **시니어 시그널** — "증상이 아니라 근본 원인을 추적했다" 류가 느껴지나
- [ ] **드릴다운 의향**이 생기나 (≥6/10)
- [ ] **신뢰 vs 의심** — 결과가 주장이 아니라 검증 가능한 성과로 읽히나

> 검증 방법: 도메인을 모르는 제3자(또는 다른 모델)를 면접관으로 두고 30초 스킴 → 위 항목을 답하게 한다.

### 9.2 통과/탈락 루브릭 (1~5점, 3점 미만이면 수정)

| 항목 | 1점(탈락) | 3점(보통) | 5점(통과) |
|---|---|---|---|
| **제목** | 작업 나열·후크·과대명명 | 토픽은 있으나 메커니즘 흐림 | 토픽 키워드 + 정직한 명명 메커니즘 |
| **대안 근거** | 결론만(재라벨링) | 버린 선택지만 언급 | 버린 선택지 + 왜 버렸나 + 채택 근거 |
| **결과 방어력** | "0"·"모든" 등 검증 불가 과장 | 방향은 맞으나 정의 모호 | 측정/상태 전환이 방어 가능 |
| **반증 가능성** | 그럴듯한 단어만 | 일부 구체 | 재현 조건·충돌 지점이 "해본 사람" 수준 |
| **비공개 일반화** | 위험 노출 or 공허 | 한쪽으로 치우침 | 기밀은 가리되 기술 실체는 구체 |
| **층 간 반복** | 같은 사실 3회 재탕 | 부분 중복 | 표/콜아웃/본문 역할 분리 |

> 6항목 평균 4점 이상 + 어느 항목도 2점 이하 없음 → 통과. 아니면 해당 항목을 고친다.

### 9.3 완성본 레퍼런스 — DnD Sibling-scoped Fractional Rank

작성·발행 완료된 첫 글이 이 골격의 **완성본 레퍼런스**다. 전문은 [`content/projects/dnd-fractional-indexing.mdx`](../../../content/projects/dnd-fractional-indexing.mdx)(중복 보존 안 함 — 정본은 그 파일). 아래는 **5층 ↔ 그 글의 실제 요소 ↔ 왜 그 문장이 필요한지**의 매핑이다. 새 글을 쓸 때 "이 층에 뭘 넣어야 하나"의 worked example로 본다.

| 층 | 그 글의 실제 요소 | 왜 이 문장이 필요한가(만족 루브릭) |
|---|---|---|
| ① 헤드라인 | 칩 `🩺 노코드 의료 문진 빌더 · DnD 순서정렬` + 제목 `5층 트리 DnD를 위한 Sibling-scoped Fractional Rank 설계` + lede 1줄 | 칩=도메인 grounding(의료 무지 독자도 5초 파악). 제목=토픽 키워드(5층 트리 DnD)+명명 메커니즘(Sibling-scoped Fractional Rank), **결과 숫자·과대명명 없음**(루브릭 제목 5점). |
| ② 요약표 PAAR | 문제(정수 순번=뒤 형제 N개 reindex) / **버린 대안·근거**(팀 정수 기본안 → 트리에 부담 → 버림) / 실행(Jira 역추적→사이값 키) / 결과(형제 rank 재계산 없음, 클라 기준) + 메타 줄 | A행에 **버린 선택지+왜**가 있어 재라벨링이 아님(대안 근거 5점). R행이 "DB 1행" 같은 과장 대신 방어 가능 표현(결과 방어력 5점). 메타 줄에 나 vs 팀·스택·비공개 고지를 **녹임**(별도 신뢰 문단 금지). |
| ③ 핵심 원인 콜아웃 | "fractional-indexing의 단순함이 새는 자리" — 형제 scope를 놓치면 같은 키 공존→정렬 비결정 | 비공개 작업물에서 **"해본 사람만 아는 구체"**가 첫 신뢰 증거(반증 가능성 5점). 제목이 '원인'이고 내용도 원인 분석(증거/원인 혼동 없음). |
| ④ 본문(지배 질문) | "팀 전원이 정수 정렬로 가는데, 왜 버렸나" — 협업·제품형 지배 질문 1개로 ADR 6축(제약→버린 선택지→채택→결과→**남은 비용**→안전 변경) 서술 | STAR 라벨 도배 대신 **케이스별 지배 질문 1개**로 판단의 질을 보임. '남은 비용(tie-break 부재·full-tree PUT)'을 빼지 않아 성공담이 아닌 ADR. |
| ⑤ 딥다이브(접지 않음) | 서술적 소제목 4개(왜 정수가 트리에서 비쌌나 / 사이값 키 / `"a0"` 충돌을 상태로 추적 / 단일 노드 변경 ≠ 단일 DB 행) | 깊이를 숨기지 않고 **의미 있는 소제목**으로 스크롤 스캔 가능. 마지막 소제목이 "정직한 경계"로 과장을 스스로 차단(비공개 일반화 5점). |

> **루브릭 자가채점 4.17/5**(6항목 평균, 2점 이하 0) — §9.2 통과. 제목/구조 게이트 + `/codex` 교차검증 후 발행(커밋 `0ac3613`).

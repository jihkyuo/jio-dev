## 0. ★ 케이스 스터디 정확성 정정 (글 쓰기 전 반드시 반영)

코드 추적 결과, 후보 백로그의 일부 framing이 **코드와 불일치**했다. 그대로 쓰면 면접관이 코드를 물었을 때 방어 불가 → 아래로 교정한다. **이게 2단계의 가장 큰 산출.**

| # | 기존 주장 | 코드 사실 | 글에서 쓸 방어 가능 표현 |
|---|---|---|---|
| **C1** DnD | "1회 = **단일 row UPDATE**" (정수 reindex는 N row) | 저장은 질문지 **트리 전체를 PUT으로 전송**. "단일 노드만 변경"은 **클라이언트 모델 한정** 사실(reorder 시 드래그된 노드 rank만 변경, 형제 불변) | "DnD가 **클라 상태에서 형제 노드를 안 건드린다**(국소 변경·낙관적 UI)"로 한정. "DB 1 row UPDATE"는 백엔드 확인 전엔 쓰지 말 것 |
| **C2** 배포 | "DRAFT/PUBLISHED/ARCHIVED **상태머신**" 본인 기여 | 상태 전이·version++는 **전부 백엔드**가 수행. FE는 enum 타입 + POST 요청 + 캐시 무효화만. **FE는 상태머신을 안 가짐** | FE 기여 = "**부수효과 동기화 전략**(낙관적 업데이트 회피 + onSettled 2캐시 무효화)". 상태머신 자체는 "백엔드와 함께 설계"인지 인터뷰로 확정 |
| **C3** 순환탐지 | "FE/BE **동형**" | **3색 DFS 부분만** 백엔드와 동형. 순서기반 후방점프 탐지는 설계문서가 "백엔드 동반은 후속"이라 명시 → **FE 단독** 가능성 | "명시 순환은 FE/BE 동형, **순서기반 후방점프는 FE가 먼저 막는 추가 방어선**"으로 분리 기술 |
| **C4** 프리뷰 | "**AI 이식**" | 코드/커밋은 "원본 앱을 미러했다"는 *결과*까지만 증명. *AI로 옮겼다*는 **커밋·주석에 전무** → 100% 인터뷰 영역 | 딥다이브 본체는 **DTO/nanoId 리팩토링** 또는 **신구토글 상태보존 설계**(코드 방어력 충분)로. "AI 협업"은 방법론 한 단락 |
| **C5** | "테스트 **167개**" | 도메인 하위 실제 테스트 파일 **196개**. 167은 백로그 헤더의 다른 제품 **커밋 수**와 혼동 | "**196개 테스트**"로 정정(더 강함) |

## 1. 딥다이브 #1 — RHF→DraftStore 동적 폼 빌더 엔진 · 중심축

### §9.1 폼 (식별자 일반화)
- **문제(실패 체인)**: [확정] React Hook Form v7 Proxy 구독 한계 — Controller 텍스트 편집(제목·설명·선택지 등)이 `formState.isDirty`에 반영 안 됨, 구조변경(setValue)만 잡힘. 근본 = watch+zodResolver+Controller 조합의 isDirty 전파 결함(RHF 공식 이슈 #4597, CLOSED·미해결). 우리 화면은 watch 30곳+ 라 정통으로 밟음. 2차 증상으로 hack 4종(resolver debounce 150ms·mount-once defaultValues·blur-trim·조건부 Provider skip) 누적. 발단은 isDirty 단독이 아니라 **영속화+undo 요구가 겹치며 "폼-제출 모델과 근본 불일치"로 전환**(설계문서).
- **재현 조건**: [확정] 이탈방지 구현 중 "문항 설명만 바꾸고 이탈하면 경고 안 뜸" 실측. 격리 재현(실 Provider): 모든 Controller 텍스트 편집 isDirty=false, 구조변경만 true. 오진 2회 후 근원 확정.
- **영향 범위(정량)**: [확정] RHF 제거 = 메인 머지 **179파일, +711/−14,172줄**("14,000줄 청산" 출처). 신규 DraftStore 빌드 189파일 +19,152. 5층 트리(Category→Question→ComponentGroup→Component→ChoiceItem). "grep 0" 검증: **도메인 슬라이스 내 `react-hook-form` import = 0**(직접 grep). 단 프로젝트 전역엔 26파일이 여전히 RHF(로그인 등 도메인 밖) → 청산 기준은 "도메인 내 0"이지 패키지 제거 아님.
- **버린 대안 + 이유**: [확정] ① RHF deep-compare 우회 — isDirty만이면 됐으나 영속화·undo 안 풀려 폐기 ② TanStack Form — "여전히 폼 패러다임" ③ Valtio — proxy 자동최적화 우아하나 예측성↓, "셀렉터 막힐 때의 카드"로 보류, PoC에서 부분구독 통과해 불필요 판정.
- **채택안 + 이유**: [확정] scoped Zustand(비전역, 문진 인스턴스마다 1회 생성) + immer + persist(localStorage) + zod + undo 미들웨어. 스택 `temporal(persist(immer(slices)))`. dirty = baseline vs 현재 deep compare로 RHF 함정 회피. immer structural sharing으로 미변경 시 reference-equality O(1).
- **검증 방식**: [확정] 도메인 테스트 **196파일**, 이 항목 직결 17파일(undo/persist/reconcile/zod/도메인룰/동등성/플로우그래프). 핵심 게이트 = **동등성 하니스**(신/구 페이로드 deep-equal "정답지 고정") + persist/undo 5층 + dangling reconcile. 청산 안전 게이트: 잎→뿌리 삭제 각 단계 tsc+테스트회계+grep 단조감소 3중. 자가 코드리뷰로 reparent rank 버그 1건 발견·픽스.
- **배포 방식**: [확정-부분] 임시 라우트(edit-v2)로 새 엔진을 프로덕션 RHF 화면 **옆에서** 키움 → 실 API 동등성 확인 → 가역 스왑 → 레거시 삭제. **스왑(가역)+삭제 2-PR 전략**. [미확인] 실제 릴리스(데스크톱 빌드/롤아웃) 타이밍은 코드에 없음.
- **결과(방어 가능)**: [확정] Before: 텍스트 편집 isDirty=false(이탈경고 누락)·hack 4종·전체 debounce 150ms. After: 텍스트 편집이 dirty를 셈·실시간 부분검증(debounce 제거)·부분구독으로 형제 행 리렌더 0. 검증 SSOT 일치: 실시간 에러와 저장차단이 **동일 도메인룰 13개 호출** → "실시간엔 보이는데 저장은 통과" 구조적 불가. 저장 실패 시 첫 에러 문항 자동 포커스+스크롤.
- **남은 비용·리스크**: [확정] persist 매 set 동기 `JSON.stringify`(5층 트리)가 입력마다 메인스레드 타격, debounce flush는 정책 문서엔 있으나 **미적용**. 이탈방지 Phase B·C 미착수. 도메인룰이 백엔드 검증 서비스를 **클라 미러링**(백엔드가 issues detail 미반환) → 백엔드 룰 변경 시 동기화 부담(구조적 후속 리스크). 도메인 TODO 3건만.
- **나 vs 팀 경계**: [확정] 도메인 전체 커밋 author 분포: **본인 575 / 팀원 4 / 팀원 1**. 스토어 디렉토리: **본인 24, 단독**. [미확인] 설계 결정의 리뷰/승인 주체, "근본 설계 제기"의 주체.
- **공개 불가**: 내부 티켓·백엔드 엔드포인트·병원/의료진 PII. 의료 사전문진 도메인 자체도 민감.

## 2. 딥다이브 #2 — DnD rank 설계 (fractional-indexing)

### §9.1 폼
- **문제(실패 체인)**: [확정] 정렬 비교가 순수 lex라 **tie-break 없음** — 같은 rank면 동순위(0) 반환 → 정렬 비결정. fractional rank는 두 노드가 같은 시퀀스 첫 항목이면 동일 문자열(`"a0"`) 가능 → tie-break 부재와 결합해 버그. invalid rank가 폼까지 흘러들면 키 생성 함수 throw → 크래시 → 입구 방어선(rank 검증·sanitize).
- **재현 조건**: [확정] **케이스 A(빈 카테고리 동일 rank)**: 빈 카테고리 2개에 각 첫 root 문항 → 둘 다 `"a0"`. cross-category 이동이 categoryKey만 cascade하고 root rank **미재발급** → 같은 카테고리에 `"a0"` 2개 → 비결정(회귀테스트 존재). [확정] **케이스 B(천장 충돌)**: 일반 선택지를 최하단 드래그 시, 맨 아래 고정된 '해당없음'(unique)보다 rank가 ≥ 되어 "일반 < 고정" 불변식 깨짐.
- **영향 범위(정량)**: [확정] rank는 5층 전 레벨 부착. reorder 액션 4종(choice/group/category/question), cross-category만 별도 액션. 케이스 A 수정 = 2파일 52줄(로직19+테스트33). [미확인] **케이스 B 티켓 전체는 답변타입 UI 작업**(NRS·해당없음 카드·3컬럼)이고 천장처리는 **1커밋 부분** → "그 티켓 = rank 버그"로 쓰면 부정확.
- **버린 대안 + 이유**: [미확인] "정수 reindex = N row → fractional 채택"이라는 비교를 **코드/커밋에서 확인 불가**. 정수→fractional 마이그레이션 흔적 없음(도입부터 fractional). → 인터뷰 필요.
- **채택안 + 이유**: [확정] `fractional-indexing ^3.2.0`. rank 계산을 한 헬퍼로 집약, 양끝 null 엣지 throw 방지. 단일 노드 변경 설계 실재(주석+코드).
- **검증 방식**: [확정] rank 키 검증(10케이스)·sanitize(8)·구조 슬라이스(17, 케이스 A 회귀가드)·경계충돌 등. 케이스 A는 RED-first 회귀가드(동일 rank 금지 단언) 후 그린. 천장 불변식 가드(drag-to-last throw 회귀).
- **배포 방식**: [미확인] 코드로 확인 불가.
- **결과(방어 가능)**: [확정-부분] "DnD 1회 = 단일 변경"은 **클라 모델 한정**(드래그 노드 rank만, 형제 불변). [미확인-반례] **네트워크/DB 단 "1 row UPDATE"는 코드와 불일치** — 저장은 트리 전체 PUT 후 form.reset. → §0 C1 표현으로. 다층 방어선(입구 sanitize·계산 집약·cross-cat 재발급·천장) 실재.
- **남은 비용·리스크**: [확정] tie-break 부재가 구조적 약점 잔존 — 케이스 A는 "동일 rank 사전 차단"이지 tie-break 추가 아님. 다른 경로로 동일 rank 생기면 재발 가능. invalid rank 흘러들 경로 실재 → 입구 sanitize가 상주 방어선(근본은 서버가 invalid rank 반환 가능, 클라가 흡수).
- **나 vs 팀 경계**: [확정] 코어 거의 단독 — reorder 슬라이스 본인 3/3, rank 키+정렬+어댑터 합산 본인 21:팀원 2. fractional 도입·케이스 A/B 픽스 모두 author 본인. [미확인] "작성자가 제안"의 *설계 발의*는 커밋이 증명 못 함(도입 커밋 author일 뿐).
- **공개 불가**: 사내 리포·내부 티켓·저장 엔드포인트·문진 도메인.

## 3. 딥다이브 #3 — 후속연결 순환탐지

### §9.1 폼
- **문제(실패 체인)**: [확정] 노드=문항, 간선=후속연결키(선택지+모든 컴포넌트). 순환 시 런타임 무한루프. **별개로**, top-level 문항은 후속키 없이 rank 순서로 암묵 진행하며 **런타임 visited 가드 없음** → 후방점프가 "정의 밖" 무한루프 생성, 일반 DFS는 순서 간선이 그래프에 없어 못 잡음. 백엔드는 순환·미해결 후속키를 400으로 막음 → FE 미검출 시 사용자는 저장 눌렀다 400 직행.
- **재현 조건**: [확정] 자기참조/직접순환/긴순환(일반 3색 DFS 케이스). **숨은 2번째 컴포넌트 간선**(한 문항 컴포넌트 2개 비-null이면 tolerant read가 첫 1개만 봐 stale 간선 누락 → 전수순회로 보완). **reorder race**(링크 시점 전방이었는데 순서변경으로 후방점프 전이, picker 판정 무효화).
- **영향 범위(정량)**: [확정] 순환탐지 모듈 356줄/순수함수 12개. 테스트 describe 9·it **33케이스**. SSOT 1모듈을 picker+저장검증(superRefine)+실시간 가시화(트리/패널/토스트)+모바일 프리뷰 진행도가 공유(소비처 10파일). [미확인] 실제 발생 빈도.
- **버린 대안 + 이유**: [확정] ① 일반 3색 DFS만 — 순서 간선 부재로 후방점프 못 잡음 → 별도 후방점프 탐지 필요. ② picker 시점 검사를 권위로 — reorder가 무효화 → "저장 시 최종상태 전체그래프 1회"를 권위로, picker disable은 조기 UX 전용. ③ 순환검사를 zod 밖 별도 패스 — canSave가 순환 못 봐 이탈모달 저장버튼 활성(400 직행) → superRefine custom issue로 주입. ④ 후방점프 탐지 초기범위 top-level 타깃 한정 → **코드리뷰에서 false-negative 판명**(child 타깃) → root 위치 기준 일반화.

**결론: 관통 테마로 둘 수 있다(코드 방어력 충분). 단 "AI가 짰다"가 아니라 "설계·검증 깊이가 AI 생산성의 상한이었다"는 축으로.**

코드가 떠받치는 근거:
- **undo/redo+immer 트러블슈팅이 단단함.** `temporal(persist(immer))` **순서**가 "임시 undo는 휘발·작성본은 영속"이라는 의도를 표현, undo set을 **leading-debounce**로 감싼 이유를 "trailing이면 한 글자만 취소 → leading이어야 입력 묶음이 한 칸 취소"까지 분해. **dangling 참조 동기화**: undo가 질문/카테고리만 되돌리고 선택 키는 별도 상태라 추적 밖 → 삭제 정책과 동일한 헬퍼로 인접 이동, 별도 커밋+테스트로 닫음. 이 경계(상태 추적 범위 ≠ UI 선택 상태)를 스스로 식별한 게 "끝까지 이해" 주장을 떠받친다.
- **검증 아키텍처가 영리함.** 실시간 에러와 저장 차단이 **동일 13룰 공유**(불일치 구조적 불가), 위반 좌표가 index 대신 key 기반이라 immer structural sharing+DnD 재정렬 후에도 매칭 안전. 동등성 하니스로 14,000줄 삭제 회귀 0을 컴파일러+테스트로 보증. **테스트 196개**.

**즉**: "AI가 타이핑했어도 작성자가 호출 체인·실패 모드·트레이드오프를 끝까지 쥐었다"는 주장은 코드로 잘 방어된다 → **관통 테마(overview에 깔고 #1 딥다이브에서 증명) 자격 있음**.

**단 주의 2가지**:
1. **"AI 이식"(#5)은 헤드라인/방법론 한 단락에 그쳐야 안전.** 코드는 "미러됨"까지만 증명하고 "AI로 옮김"은 100% 인터뷰 진술 의존 → 딥다이브 본체로 키우면 방어력 약(§0 C4).
2. **유일한 약점**: persist 성능 부채가 정책 문서엔 식별됐으나 코드 미반영 → "끝까지"의 한 칸이 의도적 보류인지 미완인지 인터뷰로만 갈림(질문 10).

**권고**: AI 네이티브를 **overview 관통 테마 + #1 딥다이브에서 코드로 증명(undo/redo·검증 SSOT·동등성 하니스)**으로 두되, 헤드라인 제목은 메커니즘(예: "RHF→스코프드 스토어 교체로 14,000줄 폼 엔진 청산")으로 가고 AI는 부제/방법론에서 받친다. case-study-structure §2 P3("명명된 메커니즘이 후크") 준수.

## 0. ★ 케이스 스터디 정확성 정정 (글 쓰기 전 반드시 반영)

코드 추적 결과, 후보 백로그의 일부 framing이 **코드와 불일치**했다. 그대로 쓰면 면접관이 코드를 물었을 때 방어 불가 → 아래로 교정한다. **이게 2단계의 가장 큰 산출.**

| # | 기존 주장 | 코드 사실 | 글에서 쓸 방어 가능 표현 |
|---|---|---|---|
| **C1** DnD | "1회 = **단일 row UPDATE**" (정수 reindex는 N row) | 저장은 질문지 **트리 전체를 PUT으로 전송**. "단일 노드만 변경"은 **클라이언트 모델 한정** 사실(reorder 시 드래그된 노드 rank만 변경, 형제 불변) | "DnD가 **클라 상태에서 형제 노드를 안 건드린다**(국소 변경·낙관적 UI)"로 한정. "DB 1 row UPDATE"는 백엔드 확인 전엔 쓰지 말 것 |
| **C2** 배포 | "DRAFT/PUBLISHED/ARCHIVED **상태머신**" 본인 기여 | 상태 전이·version++는 **전부 백엔드**가 수행. FE는 enum 타입 + POST 요청 + 캐시 무효화만. **FE는 상태머신을 안 가짐** | FE 기여 = "**부수효과 동기화 전략**(낙관적 업데이트 회피 + onSettled 2캐시 무효화)". 상태머신 자체는 "백엔드와 함께 설계"인지 인터뷰로 확정 |
| **C3** 순환탐지 | "FE/BE **동형**" | **3색 DFS 부분만** 백엔드와 동형. 순서기반 후방점프 탐지는 설계문서가 "백엔드 동반은 후속"이라 명시 → **FE 단독** 가능성 | "명시 순환은 FE/BE 동형, **순서기반 후방점프는 FE가 먼저 막는 추가 방어선**"으로 분리 기술 |
| **C4** 프리뷰 | "**AI 이식**" | 코드/커밋은 "원본 앱을 미러했다"는 *결과*까지만 증명. *AI로 옮겼다*는 **커밋·주석에 전무** → 100% 인터뷰 영역 | 딥다이브 본체는 **DTO/nanoId 리팩토링** 또는 **신구토글 상태보존 설계**(코드 방어력 충분)로. "AI 협업"은 방법론 한 단락 |
| **C5** | "테스트 **167개**" | 도메인 하위 실제 테스트 파일 **196개**. 167은 백로그 헤더의 다른 제품 **커밋 수**와 혼동 | "**196개 테스트**"로 정정(더 강함) |

## 1. 딥다이브 #1 — RHF→DraftStore 동적 폼 빌더 엔진 · 중심축

### §9.1 폼 (식별자 일반화 · ✅인터뷰 확정 반영 2026-06-27)

> 태그 추가: **[확정·IV]** = 인터뷰에서 본인 진술로 확정 · [확정] = 코드 정찰 확정 · [미확인]/[빈칸] = 동일.

- **문제(실패 체인)**: [확정·IV] **두 갈래.** ⓐ *실시간 에러 감지 마찰* — 인풋의 유효성(빈 값 필수·최대 글자수)을 **타이핑 중 즉시** 피드백(값 다 지우면 필수에러 뜸 / 에러 상태에서 채우면 즉시 해소)하려 했는데 RHF 본질상 까다로움 → 기능마다 깔끔히 안 풀리고 **외과적 끼워맞춤** 누적 → 스파게티 + 성능 저하 → 실시간 입력 트리거를 **blur로 강등**(타이핑마다 비싼 연산 버벅임)하고 **디바운스** 덧댐 → "설계가 근본부터 잘못된 것 아닌가" 의문. ⓑ *결정타* — 영속화+이탈방지 구현 시점에, 거대 폼을 localStorage 저장/복구해야 하고 에러 상태까지 복구해야 하는데 **이미 스파게티인 RHF 위에 영속화를 얹는 연산·버그 리스크**가 큼 + 이탈방지는 페이지 단위로 안 되고 **전역 상태로 올려야**(사이드메뉴/새로고침/앱종료 이탈 커버) → "RHF에 끼워맞추기보다 재정비 타이밍"으로 판단. (RHF 한계는 공식/커뮤니티 확인: [Discussion #10314](https://github.com/orgs/react-hook-form/discussions/10314)·[useFormState](https://react-hook-form.com/docs/useformstate)·[Advanced Usage](https://www.react-hook-form.com/advanced-usage/) — Controller+isDirty 루트 리렌더·controlled overhead.)
- **재현 조건**: [확정·IV] 값이 있던 인풋을 전부 지움 → 필수 에러 즉시 떠야 함 / 에러 상태에서 값 채움 → 즉시 해소돼야 함. RHF에선 이 실시간성이 깔끔히 안 잡혀 blur 강등·디바운스로 우회. (코드 정찰: 격리 재현으로 Controller 텍스트 편집이 dirty에 미반영 확인.)
- **영향 범위(정량)**: [확정] RHF 제거 = 메인 머지 **179파일, +711/−14,172줄**("14,000줄 청산" 출처). 신규 DraftStore 빌드 189파일 +19,152. 5층 트리(Category→Question→ComponentGroup→Component→ChoiceItem). "grep 0" 검증: **도메인 슬라이스 내 `react-hook-form` import = 0**. 단 프로젝트 전역엔 26파일이 여전히 RHF(로그인 등 도메인 밖) → 청산 기준은 "도메인 내 0"이지 패키지 제거 아님.
- **버린 대안 + 이유**: [확정·IV] **RHF를 그대로 둔 채 zustand+persist만 위에 얹는 외과적 최소변경안** — 변경은 최소지만 *잘못된 설계를 안고 가면 유지보수 지옥*이 될 거라 **고민 없이 폐기**. (기술적으로 쉬운 최소수술을 버리고 정도(正道)를 택함 = P4 시그널.) · [미확인·설계문서] deep-compare 우회·TanStack Form·Valtio 등 라이브러리급 후보 검토 흔적이 설계문서에 있으나 인터뷰 재확인 안 됨 — 본문 근거로 쓰려면 추가 확인.
- **채택안 + 이유**: [확정·IV] 설계단부터 근본원인 재고민 → **문진빌더에 핏한 설계 = Zustand+immer+persist+zod**. immer는 처음 사용했고 **불변객체 개념의 가치를 이때 체득**. undo/redo는 immer+zustand(zundo)로 **자연히 따라옴**(직접 구현 감 없던 걸 미들웨어로 해결). 스택 `temporal(persist(immer(slices)))` = 임시undo는 휘발·작성본은 영속.
- **검증 방식**: [확정·IV] **원칙: "믿을 수 있는 건 회귀 테스트뿐."** ① *테스트를 정답지로 고정* — RHF 시절부터 기능마다 단위+통합 테스트로 커버리지 축적, 교체 전 **누락 커버리지 점검**(테스트케이스 = 설계 변경 후 다시 그린으로 만들 타겟). ② *평행 재구현* — RHF를 **살려둔 채 밑바닥부터 신설계로 병렬 구현**(한 번에 못 바꿈/하나 건들면 전체 에러/에러 화면만 보며 재설계 불가 + 기존 구현 참고 가능 + QA 확인 가능). ③ *QA 게이트 컷오버* — 임시 진입점에서 QA가 기존과 비교 회귀 점검 → 통과 시 정본 진입점으로 승격 → **그 후** RHF 삭제. (코드 정찰: 도메인 테스트 196파일·동등성 하니스·잎→뿌리 삭제 3중 게이트로 정합.)
- **배포 방식**: [확정·IV] 위 ③이 곧 배포 전략 — 신·구 평행 운영 → **QA 통과 후 임시 진입점을 정본으로 승격 → 레거시(RHF) 제거**. [미확인] 데스크톱 앱 최종 릴리스/롤백 절차는 미확인.
- **결과(방어 가능)**: [확정] Before: 텍스트 편집 isDirty=false(이탈경고 누락)·우회 hack·blur 강등+디바운스. After: 텍스트 편집이 dirty를 셈·실시간 부분검증·부분구독으로 형제 행 리렌더 0. 검증 SSOT 일치: 실시간 에러와 저장차단이 **동일 도메인룰 13개 호출** → "실시간엔 보이는데 저장은 통과" 구조적 불가. **14k줄 삭제 회귀 0을 QA+테스트로 보증.** undo/redo 확보.
- **남은 비용·리스크**: [확정·IV] **이탈방지 미완**(현재 화면 백버튼에만 최소 구현, **사이드메뉴 이탈·앱 네이티브 이탈 미구현** — 새 설계로 가능해진 상황을 기획팀에 전달, 기획 결정 대기 중. 사이드메뉴 이탈은 본인 후속 구현 예정). [확정·IV] **검증 SSOT 미러링은 감수한 트레이드오프** — 백엔드가 말없이 검증을 바꾸면 장애지만 그건 업무방식 문제. 본인이 **FE+BE 레포를 작업영역으로 함께 보며 백엔드를 SSOT로 미러링**(백엔드 개발자에 안 물어도 됨, "항상 감시") → 부담을 cross-repo 규율로 흡수. [미확인·잠재] persist 동기 write 최적화는 정책문서에만 있고 **본인은 실사용 문제로 인지 안 함**(현재 무문제) → carried debt 아님.
- **나 vs 팀 경계**: [확정·IV] **문진빌더는 처음부터 지금까지 본인 단독 프로젝트.** 새 설계 교체 결정도 본인이 문제 분석→단독 결정. **QA만 본인+QA팀 공동 검증.** (코드 정찰: 도메인 커밋 거의 본인 단독, 스토어 디렉토리 단독.)
- **공개 불가**: 내부 티켓·백엔드 엔드포인트·병원/의료진 PII. 의료 사전문진 도메인 자체도 민감.

## 2. 딥다이브 #2 — DnD rank 설계 (fractional-indexing)

### §9.1 폼
> **✅ 인터뷰 확정(2026-06-27) — 최고 성과 + 관통 아키텍처.**
> - [확정·IV] **버린 대안 = 정수 순번(팀 전원 디폴트)**. 본인이 "트리 구조에 정수 정렬이 맞나? 재정렬 성능은?" 기시감으로 푸시백 → Jira의 대규모 순서관리(LexoRank) 역추적 → fractional-indexing 발견 → **팀에 제안해 시스템 전체 채택**. (정수→fractional 마이그 흔적 없음 = 처음부터 본인 주도 도입. §0 C2의 [미확인] 칸을 인터뷰가 채움.)
> - [확정·IV] **채택 이유**: 사이값 키 생성으로 형제 재정렬·재계산 불필요(정수 reindex의 N항목 연산 회피). "라이브러리 자체는 엣지 0, 탁월". (검증: Jira=LexoRank, fractional-indexing 동개념 — 외부 사실 확인.)
> - [확정·IV] **엣지 레이어 구분**: 라이브러리는 깔끔, *트리 통합/UI 제약*에서만 회귀 잡음(빈 카테고리 동일키 비결정 / 맨아래 고정항목 천장). → 콜아웃은 이 통합 디테일로.
> - **§0 C1 정정**: "단일 row UPDATE" ❌ → 저장은 **문진 트리 전체 PUT(설계 결정)**. 방어 표현 = **"형제 재정렬·재계산 불필요"**(rank-계산 레벨).
> - **★관통 테마 [확정·IV] — "문진 트리 전체 = 단일 데이터 단위"**: 편집 중엔 FE 상태가 정본 → 저장 시 트리 전체 PUT(델타 추적 안 함 = 시스템 복잡도 회피) → **저장 시점에만 FE+BE 이중 유효성 방어** → **export/import도 같은 트리 단위**(잘 만든 문진을 통째 JSON으로 이관) → 검증 일관성 통일·유지보수 비용↓. (DnD·검증·배포·export/import를 꿰는 축 → §4 연결. export/import 발상 출처: "복잡한 문진을 손수 만들기 어렵다 → 통째로 가져오자".)

- **문제(실패 체인)**: [확정] 정렬 비교가 순수 lex라 **tie-break 없음** — 같은 rank면 동순위(0) 반환 → 정렬 비결정. fractional rank는 두 노드가 같은 시퀀스 첫 항목이면 동일 문자열(`"a0"`) 가능 → tie-break 부재와 결합해 버그. invalid rank가 폼까지 흘러들면 키 생성 함수 throw → 크래시 → 입구 방어선(rank 검증·sanitize).
- **재현 조건**: [확정] **케이스 A(빈 카테고리 동일 rank)**: 빈 카테고리 2개에 각 첫 root 문항 → 둘 다 `"a0"`. cross-category 이동이 categoryKey만 cascade하고 root rank **미재발급** → 같은 카테고리에 `"a0"` 2개 → 비결정(회귀테스트 존재). [확정] **케이스 B(천장 충돌)**: 일반 선택지를 최하단 드래그 시, 맨 아래 고정된 '해당없음'(unique)보다 rank가 ≥ 되어 "일반 < 고정" 불변식 깨짐.
- **영향 범위(정량)**: [확정] rank는 5층 전 레벨 부착. reorder 액션 4종(choice/group/category/question), cross-category만 별도 액션. 케이스 A 수정 = 2파일 52줄(로직19+테스트33). [미확인] **케이스 B 티켓 전체는 답변타입 UI 작업**(NRS·해당없음 카드·3컬럼)이고 천장처리는 **1커밋 부분** → "그 티켓 = rank 버그"로 쓰면 부정확.
- **버린 대안 + 이유**: [미확인] "정수 reindex = N row → fractional 채택"이라는 비교를 **코드/커밋에서 확인 불가**. 정수→fractional 마이그레이션 흔적 없음(도입부터 fractional). → 인터뷰 필요.
- **채택안 + 이유**: [확정] `fractional-indexing ^3.2.0`. rank 계산을 한 헬퍼로 집약, 양끝 null 엣지 throw 방지. 단일 노드 변경 설계 실재(주석+코드).
- **검증 방식**: [확정] rank 키 검증(10케이스)·sanitize(8)·구조 슬라이스(17, 케이스 A 회귀가드)·경계충돌 등. 케이스 A는 RED-first 회귀가드(동일 rank 금지 단언) 후 그린. 천장 불변식 가드(drag-to-last throw 회귀).
- **배포 방식**: [미확인] 코드로 확인 불가.
- **결과(방어 가능)**: [확정-부분] "DnD 1회 = 단일 변경"은 **클라 모델 한정**(드래그 노드 rank만, 형제 불변). [미확인-반례] **네트워크/DB 단 "1 row UPDATE"는 코드와 불일치** — 저장은 트리 전체 PUT 후 form.reset. → §0 C1 표현으로. 다층 방어선(입구 sanitize·계산 집약·cross-cat 재발급·천장) 실재.
- **남은 비용·리스크**: [확정] tie-break 부재가 구조적 약점 잔존 — 케이스 A는 "동일 rank 사전 차단"이지 tie-break 추가 아님. 다른 경로로 동일 rank 생기면 재발 가능. invalid rank 흘러들 경로 실재 → 입구 sanitize가 상주 방어선(근본은 서버가 invalid rank 반환 가능, 클라가 흡수).
- **나 vs 팀 경계**: [확정] 코어 거의 단독 — reorder 슬라이스 본인 3/3, rank 키+정렬+어댑터 합산 본인 21:팀원 2. fractional 도입·케이스 A/B 픽스 모두 author 본인. [미확인] "작성자가 제안"의 *설계 발의*는 커밋이 증명 못 함(도입 커밋 author일 뿐).

## 7. ★ AI 네이티브 앵글 판단 — "관통 테마 vs 헤드라인만"

**결론: 관통 테마로 둘 수 있다(코드 방어력 충분). 단 "AI가 짰다"가 아니라 "설계·검증 깊이가 AI 생산성의 상한이었다"는 축으로.**

코드가 떠받치는 근거:
- **undo/redo+immer 트러블슈팅이 단단함.** `temporal(persist(immer))` **순서**가 "임시 undo는 휘발·작성본은 영속"이라는 의도를 표현, undo set을 **leading-debounce**로 감싼 이유를 "trailing이면 한 글자만 취소 → leading이어야 입력 묶음이 한 칸 취소"까지 분해. **dangling 참조 동기화**: undo가 질문/카테고리만 되돌리고 선택 키는 별도 상태라 추적 밖 → 삭제 정책과 동일한 헬퍼로 인접 이동, 별도 커밋+테스트로 닫음. 이 경계(상태 추적 범위 ≠ UI 선택 상태)를 스스로 식별한 게 "끝까지 이해" 주장을 떠받친다.
- **검증 아키텍처가 영리함.** 실시간 에러와 저장 차단이 **동일 13룰 공유**(불일치 구조적 불가), 위반 좌표가 index 대신 key 기반이라 immer structural sharing+DnD 재정렬 후에도 매칭 안전. 동등성 하니스로 14,000줄 삭제 회귀 0을 컴파일러+테스트로 보증. **테스트 196개**.

**즉**: "AI가 타이핑했어도 작성자가 호출 체인·실패 모드·트레이드오프를 끝까지 쥐었다"는 주장은 코드로 잘 방어된다 → **관통 테마(overview에 깔고 #1 딥다이브에서 증명) 자격 있음**.

**단 주의 2가지**:
1. **"AI 이식"(#5)은 헤드라인/방법론 한 단락에 그쳐야 안전.** 코드는 "미러됨"까지만 증명하고 "AI로 옮김"은 100% 인터뷰 진술 의존 → 딥다이브 본체로 키우면 방어력 약(§0 C4).
2. **유일한 약점**: persist 성능 부채가 정책 문서엔 식별됐으나 코드 미반영 → "끝까지"의 한 칸이 의도적 보류인지 미완인지 인터뷰로만 갈림(질문 10).

**권고**: AI 네이티브를 **overview 관통 테마 + #1 딥다이브에서 코드로 증명(undo/redo·검증 SSOT·동등성 하니스)**으로 두되, 헤드라인 제목은 메커니즘(예: "RHF→스코프드 스토어 교체로 14,000줄 폼 엔진 청산")으로 가고 AI는 부제/방법론에서 받친다. case-study-structure §2 P3("명명된 메커니즘이 후크") 준수.

## 9. ✅ 3단계 인터뷰 종료 — 합성 & 4단계 준비 (2026-06-27)

5개 토픽 인터뷰 완료. 각 토픽 §의 "✅ 인터뷰 확정" 콜아웃이 user-only 칸을 채움. 아래는 최종 합성.

### 9.1 §0 정정 — 인터뷰 후 최종 상태
- **C1 (DnD 단일 row)** ✅ 확정: "단일 row UPDATE" ❌ → 저장은 **문진 트리 전체 PUT(설계 결정)**. 방어 표현 = "형제 재정렬·재계산 불필요"(rank-계산 레벨). full-tree = 관통 아키텍처 테마.
- **C2 (배포 상태머신)** ✅ 갱신·강화: 단순 "BE 소유"가 아니라 **lifecycle 설계 = 문진빌더 관련자 팀 회의 공동 결정**. 본인 기여 = FE 부수효과 동기화(낙관적 회피+서버 재조회) 구현. "상태머신을 내가 설계" 주장 금지.
- **C3 (순환탐지 FE/BE 동형)** ✅ 확정: 알고리즘(3색 DFS)은 **AI 구현·본인 미추적**. 본인 소유 = 문제 식별 + 검증 전략. "정교한 알고리즘을 짰다" 주장 금지 → "순환을 제품에서 예측, 검증 전략으로 차단".
- **C4 (모바일 AI 이식)** ✅ 확정(사용자 결정): **bullet 유지**, §7 overview 증거로만. 딥다이브 본체 ❌.
- **C5 (테스트 수)** ✅ 확정: 196개.

### 9.2 §7 AI 네이티브 — 최종 (3종 양상 · §7 초안 대체)
관통 테마로 확정. "AI가 짰다"가 아니라 **"설계·검증·운전대는 내가, 가속은 AI"**. 3종 양상이 정직·뉘앙스를 만듦:
1. **깊은 이해** (토픽①): immer 처음 쓰며 불변객체·`temporal(persist(immer))`·undo/redo를 끝까지 이해(Notion 회고 7문제). "AI가 타이핑해도 끝까지 이해".
2. **스펙 위임** (토픽②): 순환 문제는 제품에서 본인이 예측·정의, 알고리즘 구현은 AI. 오너십 경계를 정직하게.
3. **계획-포팅-검수** (토픽⑤): vd-web 정본 지정·이식 계획 수립 → AI 포팅 → 통합 결함(스케일 불일치·누락) 발견·반복 수정.
+ **DnD rank**(토픽③)는 AI와 무관한 **순수 본인 설계 성과**(팀 정수 디폴트 푸시백 → Jira/LexoRank 역추적 → fractional 제안·전사 채택) → AI 네이티브 서사의 균형추(설계 역량 자체)로 둠.

### 9.3 남은 [빈칸]/[미확인] (4단계 비차단 — 글에서 일반화/생략)
- 운영 정량(이관 건수·대상 병원 수·순환탐지 실제 차단 건수·publish 사고): [빈칸] → 비공개 일반화.
- 순환탐지 relaunch revert 실제 사유: [미확인] → 알고리즘 무관(머지 처리)으로만 언급, 깊이 안 들어감.

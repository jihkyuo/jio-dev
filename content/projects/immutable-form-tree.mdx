<Chip>🩺 노코드 의료 문진 빌더 · 동적 폼 상태 설계</Chip>

의료진이 코드 없이 문진을 설계하는 빌더의 심장은, 문항·선택지가 끊임없이 추가·삭제·재배치되는 **5층 동적 트리**다. 그 트리를 받치던 React Hook Form을 걷어내고, *불변성을 아키텍처 계약으로 삼은* 상태 모델로 다시 설계한 이야기.

## 요약

| | |
|---|---|
| **문제** | RHF는 폼을 "한 번 제출하는 값 묶음"으로 본다. 그런데 이 빌더는 다섯 층이 계속 추가·삭제·재배치되고, **중간 상태를 통째로 보관(영속화)하고 되돌릴(undo)** 수 있어야 한다 — **정적 제출폼 모델과 핏이 근본부터 어긋났다.** (방아쇠: `isDirty`가 Controller 텍스트 편집을 못 잡아([#4597](https://github.com/react-hook-form/react-hook-form/issues/4597)) 저장 버튼을 변경감지로 끌 수 없었고, 늘 활성이라 "일단 저장 눌러 유효성을 반강제"하는 플로우로 전락) |
| **버린 대안·근거** | RHF를 유지하며 `isEqual`로 더티를 직접 계산하는 **우회**를 떠올렸으나 — **근본 모델이 거부하는 걸 억지로 맞추면 더 큰 부채**가 될 게 보여 코드화 전 분석에서 접었다. 트리를 자연스럽게 담는 **Zustand+immer**로 갔다. |
| **실행 (핵심 설계)** | 문진을 **key 주소로 매긴 비정규화 불변 트리**로 두고 immer로 변형해, **구조적 공유(structural sharing)가 구독·변경감지·undo의 공통 경계**가 되게 했다. 미들웨어를 `temporal(persist(immer))` 순서로 합성해 **영속과 세션 한정 undo를 구조 자체에 인코딩**. |
| **결과** | **불변성이라는 한 설계 결정**이 다섯 가지를 동시에 성립시켰다 — 노드 단위 부분 구독(형제 참조 보존)·서버 baseline 대비 변경감지·`questions/categories`만 undo 스냅샷·파생(derived) 검증·baseline 오염 없는 자가복구. "불변객체가 강력하다"가 추상이 아니라 기능으로 증명된 자리. |

## 핵심 원인 — 정적 폼 모델 위에서 동적 트리가 샌 자리

> RHF의 `mode:"onChange"`로 실시간 에러 표시 **자체는 작동했다.** 균열은 모델에 있었다 — 텍스트를 Controller로 감싸면 그 편집이 `isDirty`에 반영되지 않아([#4597](https://github.com/react-hook-form/react-hook-form/issues/4597)), **변경 여부를 알 수 없으니 저장 버튼을 끌 수가 없었다.** 늘 켜진 버튼은 "변경됐는지 모른 채 일단 저장을 눌러 유효성을 강제"하는 플로우로 이어졌다. 우회로 덧댈수록 코드는 스파게티가 됐고, 결정적으로 **거대 트리를 localStorage에 보관·복구하고 전역 이탈을 막고 undo를 얹는 요구**는 "한 번 제출하는 폼" 모델 위에서 풀 수 있는 게 아니었다. **틀린 건 RHF가 아니라, 이 문제 유형에 정적 폼 모델을 얹은 것이었다.**

## 정적 폼 모델로는 왜 안 됐고, 어떤 상태 설계로 갈았나

가장 많은 시간을 쏟은 건 기능 구현이 아니라 **설계**였다. "버그를 어떻게 고치나"가 아니라 **"이 문제 유형엔 어떤 상태 모델이 맞나"** 가 진짜 질문이었다.

**문제 유형부터 다시 봤다.** 이 빌더는 폼이 아니라 **동적 문서 에디터**에 가깝다 — 분류부터 선택지까지 다섯 층이 추가·삭제·재배치되고, 작성 중인 전체를 보관하고 되돌릴 수 있어야 한다. RHF의 제출폼 모델과는 패러다임이 다르다. (단, RHF가 나쁜 게 아니라 **이 화면의 요구에 안 맞았을 뿐** — 로그인 같은 정적 폼엔 그대로 쓰고 있다.)

**가장 쉬운 우회부터 버렸다.** `isEqual`로 더티를 직접 계산하면 RHF를 유지할 수 있었다. 하지만 isDirty는 방아쇠였을 뿐, 영속화·undo까지 더해진 요구는 그 우회로 닿지 않았다. **근본 모델이 거부하는 걸 억지로 맞추면 당장의 변경은 적어도 나중에 더 큰 부채가 된다** — 그래서 코드 한 줄 짜기 전, 분석 단계에서 접었다.

**채택한 설계의 중심은 하나다.** 문진을 **key 주소로 매긴 비정규화 불변 트리**로 두는 것. 각 노드(문항·선택지)를 배열 인덱스가 아니라 **고유 key로** 가리키고, [immer](https://immerjs.github.io/immer/)로 변형하되 안 바뀐 가지는 옛 객체를 그대로 공유한다. 이 한 결정이 이후 모든 기능의 토대가 됐다.

<Callout type="decision">
**채택 — key 주소 불변 트리.** RHF 유지를 위한 우회 대신, 폼 상태 자체를 **고유 key로 주소를 매긴 비정규화 트리**로 두고 immer로 불변 갱신했다. 핵심은 라이브러리 선택("Zustand냐 immer냐")이 아니라, **구조적 공유를 구독·변경감지·undo의 공통 경계로 삼은 것** — 한 노드만 바뀌면 그 노드의 주소만 새로 나고, 형제는 옛 주소를 유지한다.
</Callout>

**무엇이 좋아졌나 — 한 설계가 다섯을 떠받친다.** "주소가 안 바뀌면 안 변한 것"이라는 단일 원리 위에:
- 한 노드만 고치면 **그 노드를 구독한 화면만** 다시 그린다(형제는 참조가 유지돼 리렌더를 피한다)
- 변경 여부(dirty)는 **서버 baseline과 비교**하되 주소가 같은 가지는 건너뛴다(구조적 공유로 비용을 묶은 deep-compare) — 늘 켜져 있던 저장 버튼이 비로소 정확히 꺼진다
- undo는 `questions/categories`의 **과거 주소만 스택에 쌓아** 꺼낸다
- 도메인 검증은 상태로 저장하지 않고 **파생(derived)**으로 두어 stale 에러를 막는다
- 자가복구는 작성본을 살리되 **baseline은 서버 정본만** 쓰게 해 더티 기준을 오염시키지 않는다

<Callout type="note">
**감수한 트레이드오프.** 영속화는 손실 0을 우선해 매 변경을 동기로 직렬화한다 — 디바운스+flush 최적화는 후순위로 뒀다(당시 체감 이슈는 없었고, 성능을 따로 측정하진 않았다). 도메인 leaf 검증은 백엔드 룰을 **프론트에서 미러링**한다 — 백엔드가 SSOT라, 말없이 룰을 바꾸면 어긋날 수 있어 두 레포를 함께 보며 drift를 관리한다. 둘 다 "모르고 그렇게 된" 게 아니라 **기준을 정해 택한** 선택이다.
</Callout>

## 설계를 코드로 — 다섯 기능이 한 원리 위에

### 왜 동적 트리엔 제출폼 모델이 안 맞나

RHF는 폼을 "제출할 값 묶음"으로 보고, 비제어·blur 검증과 제출 시점 처리에 최적화돼 있다. 이 빌더가 원한 건 정반대다 — 다섯 층이 끊임없이 변하고, 중간 상태를 보관하고 되돌린다. 텍스트를 제어형으로 끌어오면 편집이 `isDirty`에 안 잡히고([#4597](https://github.com/react-hook-form/react-hook-form/issues/4597)), 기능마다 "이 인풋만 특별 처리"하는 우회가 쌓인다. 그 우회의 누적은 증상이고, 원인은 **모델과 문제 유형의 불일치**였다. 그래서 라이브러리를 바꾸는 게 아니라 **상태를 어떤 모양으로 들지**를 먼저 정해야 했다.

### 단일 설계 결정 — key 주소 불변 트리

핵심은 두 가지를 한꺼번에 정한 것이다. **(1) 비정규화 트리** — 문진을 서버 모델 그대로의 5층 트리로 들고, 평탄화(정규화)하지 않는다. **(2) key 주소** — 각 노드를 배열 인덱스가 아니라 고유 key로 가리킨다. 인덱스로 가리키면 드래그로 순서가 바뀔 때 "몇 번째"가 전부 틀어지지만, key는 순서가 바뀌어도 그 노드를 정확히 짚는다(검증 위반 좌표도 인덱스가 아닌 key로 매칭해, 재정렬 후에도 어떤 항목이 위반인지 안 흔들린다).

```ts
// 한 노드만 구독한다 — 형제가 바뀌어도 이 노드의 참조가 유지되면 리렌더하지 않는다
const useQuestion = (key: string) => useDraftStore((s) => s.questions[key]);

// 편집은 immer로 "직접 고치는 것처럼" 쓰지만, 결과는 불변 갱신이다
updateTitle: (key, title) =>
  set((s) => { s.questions[key].title = title; }); // 그 노드와 조상 가지만 새 객체로
```

여기에 immer를 얹으면, 한 노드를 고칠 때 **그 노드와 조상 가지만 새 객체로 나고 나머지는 옛 주소를 공유**한다(경로 복사·구조적 공유).

![문항 하나를 편집하면 그 문항→분류→문진(root)까지의 경로만 새 객체가 되고, 각 층의 형제 가지는 옛 객체를 그대로 공유한다 — 그래서 안 바뀐 가지를 구독한 화면은 리렌더되지 않는다.](/projects/immutable-form-tree-structural-sharing.svg)

### 불변성을 아키텍처 계약으로 — 한 결정이 다섯을 성립시킨다

불변성을 "좋은 습관"이 아니라 **아키텍처 계약**으로 썼다 — 각 편집이 *안정적인 정체성 경계*(주소)를 만들어내니, 다섯 기능이 모두 그 경계 위에 선다:

| 기능 | 같은 원리("주소 안 바뀌면 안 변한 것")가 어떻게 떠받치나 |
|---|---|
| **부분 구독** | 노드 selector가 key로 한 노드만 구독. 형제는 참조가 유지돼 리렌더를 피한다(참조 보존으로 불필요한 리렌더를 막는다) |
| **변경감지(dirty)** | 현재 트리를 서버 baseline과 비교 — 주소가 같은 가지는 비교를 건너뛴다(구조적 공유로 비용 제한). deep-compare를 쓰되 비교 범위가 이 공유로 묶인다 |
| **undo/redo** | 과거 트리를 스택에 쌓는 게 곧 과거 주소를 붙드는 것. 불변이라 과거가 오염되지 않는다 |
| **검증(파생)** | 검증을 상태로 저장하지 않고 selector로 파생. 입력 참조가 안 바뀌면 재계산을 건너뛴다 |
| **자가복구** | 영속본을 병합할 때 baseline만은 서버 정본으로 고정 — 작성본은 살리되 더티 기준은 오염되지 않는다 |

즉 **"불변 + key 주소 트리"라는 단 하나의 설계가** 구독·변경감지·undo·검증·복구를 *각각 따로 짜지 않고* 동시에 성립시켰다. 이게 immer를 처음 쓰며 체득한 것의 실체다 — 불변객체·프록시·경로 복사·구조적 공유라는 **언어 차원의 기본기**가, 추상이 아니라 **다섯 기능을 떠받치는 설계 레버**로 쓰인 것.

### 미들웨어 순서가 곧 제품 정책

미들웨어 합성 순서 자체가 "무엇을 되돌리고 무엇을 보관할지"를 인코딩한다.

```ts
temporal(            // 바깥: undo/redo 스택 — 영속 대상이 아니다(메모리 only)
  persist(           // 가운데: localStorage 영속 — 작성본은 새로고침·앱 종료에도 복구
    immer(slices),   // 안쪽: 5층 트리를 불변 갱신
  ),
  {
    // undo는 questions/categories만 추적, 연속 타이핑은 한 칸으로 묶는다
    partialize: (s) => ({ questions: s.questions, categories: s.categories }),
    handleSet: (fn) => debounce(fn, 400, { leading: true, trailing: false }),
  },
)
```

안쪽 immer는 트리를 불변 갱신하고, persist는 작성본을 localStorage에 남긴다. 가장 바깥 [temporal(undo 스택)](https://github.com/charkour/zundo)은 **영속 대상이 아니다** — 되돌리기 히스토리는 이번 세션 안에서만 의미가 있어 `questions/categories`만 메모리로 추적한다. undo를 leading-edge 디바운스로 묶은 건, trailing이면 연속 입력에서 마지막 한 글자만 취소되지만 leading이라야 "입력 묶음"이 한 칸으로 취소되기 때문이다. 선택 상태(어느 문항을 보는지)는 편집 상태와 섞이면 undo가 화면 포커스까지 되돌리므로 스토어 밖으로 분리했다.

### 검증을 파생으로 — 한 룰 집합을 두 경로가 공유

검증을 store 상태로 두면 모든 액션이 재검증해야 해 stale 에러가 난다. 그래서 검증을 **파생(selector)**으로 설계했다 — 한 문항에 대한 도메인 룰 13개를 순수 함수로 모아, **실시간 인라인 에러**(바뀐 문항만)와 **저장 차단**(트리 전체)이 같은 함수 집합을 호출하게 했다.

```ts
// 같은 13개 룰을 두 경로가 공유 → "화면엔 에러인데 저장은 통과"가 생길 자리가 없다
const violations = collectQuestionDomainViolations(question); // 13개 순수 함수
// 위반 좌표는 배열 인덱스가 아니라 key — 드래그 재정렬 후에도 어느 항목인지 안 흔들린다
```

이로써 "화면엔 에러인데 저장은 통과"가 생길 자리를 없앴다. (도메인 leaf 룰에 한정한 단일 출처다 — 저장은 zod 스키마, 흐름 검증은 별도 그래프가 맡는다. "모든 검증이 하나"는 아니다.)

### 노련하게 안착시키기 — 돌아가는 제품을 멈추지 않고

좋은 설계도 돌아가는 제품을 멈추면 의미가 없다. 약 14,000줄의 폼 엔진을 발견된 회귀 없이 갈아끼우는 일은 그 자체로 신중한 작업이었다. RHF를 **살려둔 채** 임시 라우트에서 새 설계를 평행으로 키우고, 새 스토어가 만든 저장 페이로드가 기존 직접 편집과 같은지 비교하는 **동등성 하니스**를 슬라이스마다 확장했다. 데이터 정합이 맞고 도메인 테스트가 전부 통과한 뒤, 도메인을 만든 내가 화면을 기능별로 육안 대조하고 QA팀이 개발 빌드를 게이트로 검증했다. 그제서야 **라우터만 가역적으로 스왑**하고, 그 다음에야 잎에서 뿌리 순으로 RHF를 제거했다(스왑과 삭제를 분리한 2개의 PR). 흥미롭게도 **처음부터 다시 구현한 덕에 기존을 더 꼼꼼히 보게 돼**, 빠져 있던 몇몇 누락을 평행 구현 도중 스스로 잡아낼 수 있었다.

### AI는 가속, 설계와 이해는 내가

핵심 설계의 후보는 AI가 제안했다. 하지만 **그 제안 안에서 비교·검증·확정한 것은 나였고**, 생소한 immer를 핵심 엔진에 넣는 결정 앞에서 토대 개념(참조·불변·구조적 공유·GC)까지 직접 파고들었다. 그 이해가 있었기에 — 형제 참조가 유지되면 리렌더를 피한다는 걸 알고 부분 구독을 설계했고, undo 스택이 참조로만 과거를 붙든다는 걸 알았기에 선택 상태가 추적 밖이라 dangling되는 문제를 미리 막을 수 있었다. **AI가 타이핑을 가속해도, 설계와 검증의 깊이가 그 가속의 상한이었다.** 시니어로 갈수록 자료구조·언어 기본기가 왜 중요한지를, 이 작업이 가르쳐줬다.

### 설계가 열어준 길 — 전역 이탈 방지

새 설계의 진짜 수확 하나는, **이전 구조에서는 시도조차 못 하던 일이 가능해졌다는 것**이다. RHF 시절엔 상태가 페이지 단위라 사이드메뉴 이동·새로고침·앱 종료 같은 전역 이탈을 막을 길이 없었다. 상태를 인스턴스 단위로 끌어올리고 영속화를 깔자, 비로소 **작성 중인 내용을 잃지 않게 이탈을 가로채는** 설계가 가능해졌다. 현재 화면 백버튼 단계를 1차로 구현했고, 메뉴 이탈·OS·앱 종료 가드는 같은 토대 위에서 후속으로 확장할 수 있게 됐다(일부는 기획 결정 대기 중). 미완의 고백이 아니라, **설계 변경이 새 기능의 길을 연** 자리다.

의료진이 코드 없이 문진을 설계하는 빌더를, 화면 스펙이 빈약한 채로 영업일 10일이 안 되는 마감에 밀어넣어야 했다. 원 계획 안에서는 사람이 스펙을 읽고 손으로 구현하는 방식이 현실적이지 않았다. *스펙을 프로젝트에 내장하고 테스트를 회귀 방화벽 삼은* 개발 하네스로, 구현을 AI에 맡긴 이야기.

## 핵심 원인 — 사람이 병목이 되는 자리

> PRD는 버전업을 멈추지 않는데, 사람이 매번 그걸 다시 읽는 속도는 AI가 코드를 쏟아내는 속도를 못 따라간다. 그 격차가 **스펙 드리프트**다 — AI는 구식 스펙으로 구현하고, 사람은 그걸 뒤늦게 발견한다. 빈약한 Figma까지 겹치면 사람이 스펙을 해석해 전달하는 경로 전체가 느려진다. 그래서 문제를 "AI가 더 잘 짜게 하기"가 아니라 **"AI가 늘 최신 스펙을 보게 하고, 어긋나면 바로 잡히게 하기"**로 다시 정의했다 — 스펙을 사람이 읽는 문서가 아니라 프로젝트가 드는 자산으로 옮기고, 드리프트 감지를 작업 루프 안에 넣었다.

## 왜 AI를 보조가 아니라 운전대에 앉혔나 — 그리고 어떻게 안전하게 했나

불가능한 마감 앞에서 실질적 선택지는 좁았다. 스펙 구멍을 붙들고 논의로 시간을 태우거나, AI에 구현을 맡기되 **그 산출물을 신뢰 가능하게 만드는 장치**를 먼저 세우거나. 문진빌더 이전까지 나는 AI를 자동완성 수준 보조로만 썼다. 이번엔 그 자세로는 규모를 못 쳐낸다는 게 분명했다.

그래서 AI를 운전대에 앉히되, **AI가 딴 데로 새지 않게 붙잡는 하네스**를 먼저 세웠다. 하네스(harness)는 힘을 엉뚱한 방향이 아니라 온전한 방향으로 쏟게 하는 장치다. 여기서 힘은 AI의 구현 속도였고, 붙잡는 줄은 세 가닥이었다 — **내장된 스펙**, **회귀 방화벽이 된 테스트**, **프론트·환자웹·백엔드 저장소를 함께 읽는 크로스레포 컨텍스트**.

<Callout type="decision">
**채택 — 하네스로 AI의 폭발반경을 묶는다.** 스펙을 프로젝트에 내장해 AI가 추적하고 드리프트를 세션마다 감지한다(구멍은 기능 플로우에 녹여 구현하고 후속으로 정합). AI가 구현 초안을 대량 생성하되, 테스트가 스펙이자 회귀 방화벽이 되어 **인접 기능에서 생기기 쉬운 회귀를 빌드에서 막는다.** 도메인 규칙은 한 곳에 모으고, 문항·선택지를 고유 key로만 가리키게 해, 생성된 코드가 검증과 주소 체계를 제각각 흔들기 어렵게 만들었다. "AI가 짰다"가 아니라 **"AI가 짜도 어긋남이 빨리 잡히는 판을 내가 깔았다"**가 핵심이다.
</Callout>

**무엇이 좋아졌나.** 영업일 10일 미만이라는, 규모상 불가능한 계획이 3주 풀타임 개발 + 4주차 QA 병행으로 실제 인도됐다. 인도한 것은 데모가 아니라 프로덕션 ~29k LOC와 그와 맞먹는 ~27k LOC의 테스트(167파일·약 1,159 케이스)였다 — 프로덕션과 비슷한 규모의 테스트가 깔려 있어, AI가 만든 변경도 빌드 안에서 계속 검증됐다. 더 중요한 건 복리였다. 이때 쌓인 커버리지가 지표가 되어, 이후 폼 [상태 엔진을 통째로 재설계](/projects/immutable-form-tree)할 때 그 위험한 작업에 가속을 붙여줬다.

<Callout type="warning">
**남은 비용·후속 리스크.** 속도를 위해 구멍을 기능에 녹여 구현하고 정합을 뒤로 미룬 만큼, **정합 백로그가 실제 비용으로 돌아왔다.** 이건 AI의 대가라기보다 기획 속도가 개발 속도를 못 따라온 결과에 가깝지만, 빠르게 가면 남기 쉬운 부채다. 기술 부채보다 더 큰 숙제는 조직 쪽이었다. 개발자가 기획·디자인 공백까지 흡수해 속도를 내면, 기존의 기획→디자인→개발→QA 순차 흐름은 오히려 병목이 된다. 결국 남은 비용은 **팀의 역할과 워크플로우를 다시 맞추는 일**이었다.
</Callout>

## 스펙을 사람이 아니라 프로젝트가 든다

기획자는 PRD를 계속 고도화했지만, 그 문서는 사람이 통독하기엔 도메인 용어가 빽빽하고 Figma와도 어긋났다. 사람이 읽어 구현으로 옮기는 경로는 버전업마다 다시 느려진다. 그래서 스펙을 **프로젝트에 내장된 자산**으로 두고 AI가 추적하게 했다. 원본 PRD는 Confluence에 있었다. 여기에 **[Confluence MCP 서버](https://modelcontextprotocol.io)**를 붙였다. MCP(Model Context Protocol)는 AI가 Confluence 같은 외부 도구를 읽게 해주는 연결 규약이다. 덕분에 AI는 사람이 복사해 준 요약본이 아니라 Confluence 원본 페이지를 직접 확인할 수 있었다. 프로젝트에 내장한 스펙 사본마다 원본 페이지의 버전·확인일 같은 메타를 달아 두면, 세션 시작 루틴이 그 메타를 보고 **싱크 스킬**을 돌린다. 스킬은 Confluence MCP로 원본의 현재 버전을 당겨와 내장 사본과 비교하고, 오래됐으면 차이를 사람의 승인 아래 갱신한다. 사람은 스펙을 통독하는 역할에서 내려오고, "이 스펙이 지금 코드와 맞나"를 판정하는 역할로 올라간다.

빈약한 화면은 논의로 메우지 않고 기능 플로우에 녹여 구현했다 — 어디를 자체 기획으로 메웠는지는 별도 문서에 "정책 미정"으로 추적해, 개발 이후 기획과 다시 맞출 목록으로 남겼다. 속도를 얻는 대신 정합을 백로그로 이연한, 의도된 트레이드오프다.

## 테스트를 회귀 방화벽으로

AI가 A 기능을 짜다 B를 조용히 깨는 일은 상시적이다. 하네스라는 말을 알기 전부터, 나는 **테스트가 그런 회귀를 조기에 드러내는 장치**라고 봤다. 테스트 케이스는 그 자체가 기능의 스펙이자 동작 정의라서, 인접 변경이 스펙을 어기는 순간 빌드가 걸린다. 처음엔 구현 뒤 테스트를 보강했지만, 곧 **먼저 쓰는 편이 구현 중에 더 정밀한 품질을 낸다**는 걸 스스로 깨달았다.

그래서 이 도메인은 테스트가 프로덕션 코드와 맞먹는 밀도로 깔려 있다(167파일·약 1,159 케이스). 방화벽이 지킨 경계는 구체적이다. 백엔드가 깨진 순서 키를 돌려줘도 "답변 추가"에서 키 생성이 터지지 않게 막았다. 답변 타입을 바꿀 때 선택지 메타가 사라지지 않는지도 신·구 저장 결과를 대조해 확인했다. 저장 버튼과 이탈 모달이 겹쳐 저장 요청이 두 번 나가는 레이스도 테스트로 막았다. 이건 "AI가 이 버그를 잡았다"는 무용담이 아니라, **하네스가 상시 지키는 경계**의 목록이다.

여기에 검증을 한 곳으로 모으는 규율을 얹었다. 실시간 에러와 저장 차단이 [같은 도메인 규칙 묶음을 공유](/projects/immutable-form-tree)하게 해, "화면엔 에러가 보이는데 저장은 통과"하는 어긋남이 생길 가능성을 구조적으로 줄였다. 생성된 코드가 UI마다 검증을 제각각 다시 구현할 여지를 좁힌 것이다.

## 백엔드까지 한 작업영역에 — API 명세 대기를 코드 확인으로

가장 흔한 병목 하나는 백엔드 개발자와의 API 명세 소통이다. 프론트·환자용 웹·백엔드 저장소를 **한 작업영역으로 묶어** AI가 전 저장소를 탐색하게 하자, AI가 백엔드 코드를 직접 읽고 DTO의 실제 모양을 파악했다. 명세를 사람에게 물어 기다리던 왕복이, 코드를 직접 확인하는 일로 옮겨가며 그 대기가 줄었다.

![전에는 FE가 백엔드 개발자에게 API 명세를 물어 기다렸지만, 후에는 프론트·환자웹·백엔드를 한 작업영역에 묶어 AI가 백엔드 코드를 직접 읽어 DTO를 파악해 명세 대기를 줄였다.](/projects/ai-native-harness-api-bottleneck.svg)

## 설계는 나, 가속은 AI — 개발자 역할이 바뀐 지점

이 글은 두 편의 딥다이브 위에 선다. 폼 [상태 엔진을 불변 트리로 재설계한 글](/projects/immutable-form-tree)은 AI가 설계 후보를 제안해도 그것을 비교·검증·확정한 깊이가 나에게 있었음을 보인다(AI 협업에서 깊은 이해의 몫). [트리 DnD 순서 설계 글](/projects/dnd-fractional-indexing)은 AI 없이 팀의 기본안을 뒤집고 순서 시스템을 제안·관철한, 순수한 설계 역량의 균형추다. 하네스는 그 둘을 꿰는 상위 서사다.

경계는 정직하게 긋는다. 구현 초안과 설계 후보는 AI가 대량으로 냈지만, **문제 정의·후보 검증·정본 판단·회귀 방어선은 내 몫**이었고, 그 깊이가 AI 가속의 상한이었다. 하네스의 필요성·구성·운영 규율을 설계하고 도입한 것도 나다. 이 경험으로 개발자의 자리를 다시 봤다 — 코드를 손으로 유지보수하는 사람에서, **AI가 신뢰 있게 코딩하도록 하네스를 유지보수하고 설계자와 AI를 오케스트레이션하는 사람**으로. 그리고 AI가 곧장 구현에 뛰어들기 전에 먼저 계획을 세우고 검증하게 하는 것이, 속도와 안전을 동시에 얻는 길이라는 것도. 지금은 이 하네스를 더 나은 형태로 다듬는 법을 학습해 바로 프로젝트에 적용하고, 팀에도 전파하고 있다.

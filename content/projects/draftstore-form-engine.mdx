<Chip>🩺 노코드 의료 문진 빌더 · 폼 상태 엔진</Chip>

의료진이 코드 없이 문진을 설계하는 빌더의 심장은 5층 트리를 편집하는 거대한 동적 폼이다. 그 폼을 받치던 React Hook Form을 걷어내고, *실시간 검증·영속화·되돌리기를 한 설계 안에 담는* Zustand+immer 스토어로 다시 지은 결정.

## 요약

| | |
|---|---|
| **문제** | 텍스트 인풋을 Controller로 감싸면 그 편집이 **`isDirty`에 잡히지 않았다**(RHF 이슈 [#4597](https://github.com/react-hook-form/react-hook-form/issues/4597)). 값을 다 지워도 더티로 안 잡혀 이탈 경고가 새고, 이를 막으려 끼운 우회 코드가 한 뿌리에서 계속 증식했다. 결정적으로, 거대 트리를 localStorage에 **영속화**하고 **undo/redo**까지 얹어야 하는 요구가 겹치면서 RHF의 폼-제출 모델 자체가 한계로 드러났다. |
| **버린 대안·근거** | 1차안은 **`isEqual`로 더티를 직접 계산하는 deep-compare 우회** — isDirty 버그만이면 RHF를 그대로 둘 수 있었다. 하지만 **영속화·undo 요구는 이 우회로 풀리지 않았다**(폼-제출 모델과 근본 불일치). 라이브러리 후보(TanStack Form·Valtio·Jotai)도 "폼 패러다임" 또는 예측성 문제로 버리고, 트리에 자연스럽고 스코프 생명주기가 명확한 **Zustand+immer**를 택했다. |
| **실행** | 폼 상태를 9개 슬라이스로 쪼갠 **`temporal(persist(immer))`** 스토어로 재설계. undo 히스토리는 메모리에서 휘발하고 작성본은 영속한다. RHF를 **살려둔 채 임시 라우트에서 평행 재구현**해 동등성을 맞추고, **라우터만 스왑**(가역)한 뒤 잎에서 뿌리 순으로 RHF를 들어냈다. |
| **결과** | 텍스트 편집이 **dirty를 정확히 세고**(이탈 경고 누락 해소), 실시간 검증과 저장 차단이 **같은 13개 도메인 룰**을 재사용해 디바운스 자체가 불필요해졌다. 슬라이스 부분 구독으로 형제 행 리렌더가 사라졌다. **약 14,000줄을 삭제한 뒤 발견된 회귀 0**으로 QA·테스트 게이트를 통과했고, undo/redo도 확보. |

## 핵심 원인 — isDirty가 텍스트 편집을 놓친 자리

> RHF의 `mode:"onChange"`로 실시간 에러 표시 **자체는 작동했다.** 균열은 다른 곳이었다 — 텍스트 인풋을 Controller로 감싸면 그 편집이 `isDirty`에 반영되지 않는다(RHF 이슈 #4597). 그래서 값을 다 지워도 폼이 "변경 없음"으로 보고, **이탈 경고와 저장 판정이 어긋났다.** 이걸 막으려 `onChange` 전체 검증 비용을 디바운스로 가린 리졸버, blur에서 값을 다듬는 처리, mount-once 기본값 참조, 조건부 Provider 스킵 같은 우회가 쌓였는데 — **전부 "RHF에 안 맞는 걸 억지로 얹은" 한 뿌리에서 나온 증상이었다.**

## 한 번에 갈아엎지 않고 14,000줄을 어떻게 들어냈나

가장 어려운 판단은 "RHF를 무엇으로 바꾸나"가 아니었다. **돌아가는 제품의 폼 엔진을 회귀 없이 통째로 교체하는 방법**이었다.

**왜 어려웠나.** isDirty 버그는 방아쇠였을 뿐, RHF를 버리는 진짜 이유는 **동적 5층 트리 + 영속화 + undo**를 폼-제출 모델 위에 얹는 근본 불일치였다. 이 폼은 분류→문항→그룹→컴포넌트→선택지를 편집하는 약 14,000줄짜리 엔진이고, 빌더의 문진 편집 화면 전체가 여기에 매달려 있다. 하나를 건드리면 전체가 에러를 뱉어 "에러 화면만 보며 재설계"하는 건 불가능했다.

**가장 쉬운 길을 먼저 버렸다.** 처음 검토한 건 `isEqual`로 더티를 직접 계산하는 deep-compare 우회였다. isDirty 문제만이라면 이걸로 RHF를 유지할 수 있었다. 하지만 거대 트리를 localStorage에 저장·복구하고 세션 밖에서 undo를 얹는 요구는 이 우회로 풀리지 않았다 — 폼을 "한 번 제출하는 값"으로 보는 모델과, 폼을 "계속 편집·보관·되돌리는 문서"로 보는 요구가 근본부터 달랐다. 라이브러리 후보를 비교한 뒤(TanStack Form은 여전히 폼 패러다임, Valtio는 예측성, Jotai는 동적 트리 모델링 부담) 트리를 자연스럽게 담고 스코프 생명주기가 명확한 Zustand+immer로 갔다.

<Callout type="decision">
**채택 — immer 기반 슬라이스 스토어.** RHF 유지를 위한 deep-compare 우회 대신, 폼 상태 자체를 9개 슬라이스로 쪼갠 Zustand+immer 스토어로 재설계했다. 미들웨어 합성 순서 하나로 영속과 세션 한정 undo/redo를 동시에 얻고, 실시간·저장 검증을 한 룰 집합에 묶었다. (합성 순서가 왜 정책인지는 딥다이브에서.)
</Callout>

**무엇이 좋아졌나.** 텍스트 편집이 더티를 정확히 세어 이탈 경고 누락이 사라졌고, 실시간 검증이 우회 없이 동작한다. 슬라이스 부분 구독으로 한 행을 고쳐도 형제 행은 리렌더되지 않는다. 무엇보다 **실시간 에러 표시와 저장 차단이 같은 13개 도메인 룰을 재사용**하게 묶어, "화면엔 에러가 보이는데 저장은 통과되는" 모순이 생길 자리를 없앴다.

**어떻게 안전하게 바꿨나.** 세 겹의 게이트로 회귀를 막았다. ① **테스트를 정답지로 고정** — RHF 시절부터 기능마다 단위·통합 테스트를 쌓아왔고(도메인 테스트 196개 파일), 교체 전 누락 커버리지를 점검해 "재설계 후 다시 초록으로 만들 타겟"으로 삼았다. ② **평행 재구현** — RHF를 지운 게 아니라 **살려둔 채** 임시 라우트에서 새 스토어를 키워, 같은 API·같은 데이터 경로로 동등성을 맞추고 QA가 둘을 비교할 수 있게 했다(스토어 페이로드가 직접 편집과 일치하는지 검사하는 동등성 하니스). ③ **라우터 스왑 컷오버** — 한 화면은 하나의 출처라 부분 공존이 안 되므로, 전체 검증을 통과한 뒤 **라우터만 가역적으로 스왑**하고, 그 다음에 잎에서 뿌리 순으로 RHF를 삭제했다(스왑과 삭제를 분리한 2개의 PR).

<Callout type="warning">
**남은 비용·후속 리스크.** **이탈 방지가 아직 절반이다** — 현재 화면 백버튼 단계만 들어갔고, OS·앱 종료 가드와 사이드메뉴·로그아웃 가드는 미착수라 기획 결정을 기다린다. **영속화는 손실 0을 우선해 매 변경마다 동기로 직렬화**하는데, 디바운스+flush 최적화는 후속으로 남겼다(현재 실사용 체감 문제는 없음). 그리고 검증 룰을 백엔드에 **미러링**한 건 감수한 트레이드오프다 — 다만 단순 복제가 아니라 **프론트가 더 엄격한 방향**으로 두어(예: 백엔드는 0개만 차단, 프론트는 2개), 프론트 통과는 곧 백엔드 통과라 저장 단계에서 막히지 않는다. 백엔드가 룰을 바꾸면 어긋날 수 있어 두 레포를 함께 보며 흡수한다.
</Callout>

## 딥다이브

### 왜 RHF에 이 폼이 안 맞았나

RHF는 폼을 "한 번 제출하는 값 묶음"으로 본다. 비제어·blur 검증에 최적화돼 있고, 제출 시점에 검증·전송하는 흐름에 강하다. 이 빌더가 원한 건 정반대였다 — 분류부터 선택지까지 다섯 층이 끊임없이 추가·삭제·재배치되고, 그 중간 상태를 통째로 보관하고 되돌릴 수 있어야 했다. 텍스트를 Controller로 끌어와 제어형으로 다루면 편집이 `isDirty`에 반영되지 않고(이슈 #4597), 루트 리렌더와 제어 컴포넌트 오버헤드가 따라붙는다. 기능마다 "이 인풋만 특별 처리"하는 우회가 쌓인 건 증상이었고, 진짜 원인은 모델의 불일치였다.

### 버린 길 — deep-compare 우회와 라이브러리 후보

가장 적게 바꾸는 길부터 저울질했다. `isEqual(defaultValues, getValues())`로 더티를 직접 계산하면 RHF의 내부 더티 로직을 우회해 isDirty 버그를 덮을 수 있었다. 거기서 멈추지 않은 이유는 요구가 isDirty 하나가 아니었기 때문이다 — localStorage 자동 저장·복구와 undo는 "제출 모델"을 우회로 덧대는 걸로는 닿지 않았다. 라이브러리급 후보도 함께 봤다. TanStack Form은 fine-grained 구독이 RHF보다 낫지만 여전히 폼 패러다임이고, Valtio는 proxy 자동 최적화가 트리에 우아하나 예측성이 떨어지며, Jotai는 동적 atom 트리 모델링이 부담이었다. 결론은 명시적이고 트리를 그대로 담으며 인스턴스별 생명주기가 분명한 Zustand+immer였다.

### `temporal(persist(immer))` — 휘발하는 undo, 영속하는 작성본

미들웨어 합성 순서가 곧 정책이다. 안쪽 immer는 9개 슬라이스가 5층 트리를 직접 변형하게 하고(불변 갱신은 immer가 뒤에서 처리), 그 위 persist는 작성 중인 문진을 localStorage에 남겨 새로고침·앱 종료에도 복구되게 한다. 가장 바깥 temporal(undo/redo 스택)은 **영속 대상이 아니다** — 되돌리기 히스토리는 이번 세션 안에서만 의미가 있어 `questions`·`categories`만 메모리로 추적하고 디스크로 내보내지 않는다. undo는 400ms leading-edge 디바운스로 묶었다. trailing이면 연속 입력에서 마지막 한 글자만 취소되지만, leading이라야 "입력 묶음"이 한 번에 한 칸씩 취소돼 사람이 기대하는 되돌리기가 된다. 선택 상태(어느 문항을 보고 있는지)는 편집 상태와 섞이면 undo가 화면 포커스까지 되돌리므로 **스토어 밖으로 분리**했다.

### 검증을 한 곳에서 — 13개 도메인 룰과 비대칭 미러링

이 스토어에서 가장 영리하게 푼 지점이다. 한 문항에 대한 도메인 검증을 13개 순수 함수로 모아(`collectQuestionDomainViolations`), **실시간 인라인 에러**(변경된 문항만 selector로 재검증)와 **저장 차단**(트리 전체를 zod로 1회 검증)이 같은 함수 집합을 호출한다. 두 경로가 룰을 따로 들면 언젠가 어긋나 "화면엔 에러인데 저장은 통과"가 생기는데, 단일 출처로 묶으니 그 불일치가 들어설 자리가 없다 — 부분 검증이 문항 단위라 RHF 시절의 디바운스 리졸버도 통째로 불필요해졌다. 위반 좌표는 배열 인덱스가 아니라 **키 기반**이라, immer의 구조적 공유와 [드래그 재정렬](/projects/dnd-fractional-indexing)로 노드 위치가 바뀐 뒤에도 어떤 항목이 위반인지 정확히 매칭된다. 이 룰들은 백엔드 검증을 미러링하되 **프론트가 더 엄격한 방향**으로 둬서, 프론트를 통과하면 백엔드에서 막히지 않는다.

### 영속화의 정직한 복구 — baseline은 서버에서

localStorage 복구에는 함정이 있다. 백업을 그대로 믿으면, 더티 판정의 기준이 되는 원본(baseline)까지 오래된 백업으로 덮여 "변경 안 했는데 변경됨"이 된다. 그래서 영속화 대상에서 **baseline을 제외**하고 진입 시 항상 서버 정본에서만 받게 했다. 복구는 백업을 바로 쓰지 않고 서버 정본과 대조하는 자가복구 게이트를 거친다 — 작성 중이던 내용은 살리되, 무엇이 바뀌었는지의 기준선은 서버가 쥔다. "에러 상태까지 보관하는 영속화"와 "오염되지 않는 더티 기준"을 동시에 만족시키는 자리다.

### 살려둔 채 다시 짓기 — 평행 재구현과 컷오버

약 14,000줄을 들어내는 일에서 "회귀가 없었다"는 말은 검증 장치 없이는 빈말이다. 그래서 RHF를 지우는 대신 임시 라우트에 새 스토어를 평행으로 세웠다. 기존 테스트가 정답지였고(도메인 테스트 196개 파일), 새 구현이 그걸 전부 초록으로 통과하는 게 컷오버의 조건이었다. 스토어가 만든 저장 페이로드가 직접 편집과 동일한지 비교하는 동등성 하니스를 슬라이스마다 확장했다. 동등성을 맞춘 뒤 QA가 임시 화면과 기존 화면을 대조하고, 통과 후 **라우터만 가역적으로 스왑**, 그 다음에야 잎에서 뿌리 순으로 레거시를 제거했다(스왑 PR과 삭제 PR을 분리). 이 작업은 AI 협업으로 빠르게 갔지만, 빠른 게 위험하지 않았던 건 호출 체인·실패 모드·트레이드오프를 끝까지 쥐고 있었기 때문이다. immer를 처음 쓰면서도 불변 갱신이 어디서 새는지, undo 스택이 무엇을 추적하고 무엇을 놓치는지를 직접 따라갔다 — AI가 타이핑을 가속해도 설계와 검증의 깊이가 그 가속의 상한이었다.

### 정직한 경계 — 이탈 방지는 아직 절반

새 설계의 동기 중 하나는 전역 이탈 방지였는데, 그 약속을 다 지키지는 못했다. 지금 막는 건 현재 화면의 백버튼 단계뿐이고, OS·앱 종료 가드와 사이드메뉴·로그아웃 가드는 미착수다. 다만 이건 **이전 RHF 구조에서는 시도조차 어려웠던** 것이고, 새 스토어가 상태를 인스턴스 단위로 들고 있으니 이제 가능해진 영역이다 — 남은 단계는 동작 정의가 필요해 기획 결정을 기다리고 있다.
